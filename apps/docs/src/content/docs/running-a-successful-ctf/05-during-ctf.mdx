Now that you've deployed the challenges and the CTF platform is up and running, it's time to start the competition. This section covers monitoring practices, incident response procedures, and operational guidelines for handling issues that arise during the event.

## Monitoring

Effective monitoring is essential for identifying and resolving issues before they significantly impact participants.

### Infrastructure monitoring

Monitor the following components throughout the competition:

- **rCTF platform** - Watch API server logs for elevated error rates.
- **PostgreSQL** - Monitor connection pool usage, query latency, and disk space. High connection counts may indicate a connection leak or excessive load.
- **Redis** - Monitor memory usage and connected clients. Redis is used for caching, rate limiting, and leaderboard computation. If Redis becomes unavailable, rate limiting and caching will fail.
- **Challenge containers** - Monitor CPU, memory, and network usage for hosted challenges.
- **Reverse proxy (Nginx)** - Monitor request rates, error rates (5xx), and response times. If behind Cloudflare, use the Cloudflare dashboard for DDoS metrics.

<Callout variant="tip">

Consider setting up a simple uptime monitor (e.g., [UptimeRobot](https://uptimerobot.com/), [Hetrix Tools](https://hetrixtools.com/)) that pings your CTF platform and alerts via Discord or email if it goes down.

</Callout>
### Staffing and shift coverage

For competitions exceeding 12 hours, it is advisable to establish a shift rotation to ensure continuous coverage. When possible, leverage team members in different time zones to provide around-the-clock support without requiring overnight shifts. At minimum, ensure that at least one person with infrastructure access is available at all times to respond to critical incidents. If authors are unavailable for their specific challenges, ensure adequate internal documentation is available for a support team member to help with specific tickets.

## Incident response

Despite thorough preparation, issues will inevitably arise during the competition. Establish clear procedures for common incident types.

### Challenge outages

If a challenge becomes unavailable, the response depends on the deployment method:

- **Docker containers** - Check container status with `docker ps{:sh}` and logs with `docker logs <container>{:sh}`. Restart with `docker compose up -d --force-recreate <service>{:sh}` if needed.
- **Instancer-managed challenges** - Verify the instancer service is running and reachable. Check the instancer logs for errors. If individual instances are failing, the issue may be with the challenge image itself rather than the instancer.
- **Static challenges** (no remote) - Verify that challenge files are accessible through the upload provider. If using S3/GCS, check bucket permissions and CDN status.

Once the issue is resolved, post an announcement in the `#announcements` channel informing participants that the challenge is back online.

### Platform issues

If the rCTF platform itself becomes unresponsive:

1. **Check API server logs** - Look for crash traces, out-of-memory errors, or unhandled exceptions. If using Docker: `docker logs rctf-app-1{:sh}`.
2. **Verify database connectivity** - Ensure PostgreSQL is running and accepting connections. Check for connection pool exhaustion or long-running queries: `docker exec -it rctf-postgres-1 psql -U rctf -c "SELECT count(*) FROM pg_stat_activity;"{:sh}`.
3. **Verify Redis connectivity** - Ensure Redis is running: `docker exec -it rctf-redis-1 redis-cli ping{:sh}`.
4. **Restart the rCTF service** - If the issue is not immediately identifiable: `cd /opt/rctf && docker compose restart app{:sh}` (or the relevant service name).
5. **Check resource usage** - Verify the host has sufficient CPU, memory, and disk space. High leaderboard update frequency or large team counts can increase resource usage.

For extended outages, consider extending the competition end time to compensate for lost time.

### Security incidents

Security incidents require immediate attention. Common scenarios include:

- If evidence of **flag sharing** between teams is identified, document the evidence and consider disqualification.
- If challenge infrastructure is being targeted (e.g., DoS attacks), consider temporarily taking the affected challenge offline and implementing additional rate limiting or IP blocking.
- If a flag is accidentally leaked (e.g., in a public announcement or challenge description), the flag must be rotated immediately. Update the challenge configuration in rCTF and redeploy any affected remote services.

## Challenge issues

### Broken challenges

If a challenge is reported as broken, the first step is verification. Have the challenge author test the intended solution against the live deployment. If the challenge is confirmed broken:

1. Post an announcement acknowledging the issue and indicating that a fix is in progress.
2. Apply the necessary fix to the challenge.
3. Test the fix thoroughly before redeployment.
4. Announce when the challenge is operational again.

<Callout variant="warning" title="Mid-competition modifications">

Exercise extreme caution when modifying challenges during the competition, as changes can invalidate progress made by participants. If any team has already solved the challenge, the challenge should generally be considered solvable and modifications should be avoided unless the challenge is fundamentally broken.

</Callout>
### Unintended solutions

Unintended solutions (where participants solve a challenge through a method not anticipated by the author) are common in CTFs. The recommended approach is to accept the unintended solution as valid, as penalizing creativity would be unfair to participants who invested effort into finding the alternative approach.

If the unintended solution trivializes the challenge (e.g., a configuration error exposing the flag directly), consider deploying a "revenge" variant with the vulnerability patched. Note that releasing a revenge challenge may inadvertently hint at the nature of the unintended solution.

### Attachment updates

If challenge attachments require updates (e.g., to fix typos or provide missing files), update the attachment and post a clear announcement specifying what changed. Avoid changes that would invalidate existing progress.

## Hint policy

As established in the [prerequisites section](/running-a-successful-ctf/01-prerequisites/#hints), a strict no-hint policy is recommended for fairness. However, structured hint releases may be appropriate for unsolved challenges as the competition progresses. If hints are to be released, follow this procedure:

1. Around the halfway point of the competition, request that interested participants open support tickets describing their current progress on unsolved challenges.
2. The challenge author reviews submitted progress and determines whether a hint is warranted and, if so, what information to provide.
3. Any approved hints must be released publicly in the `#announcements` channel to ensure all participants have equal access.

<Callout variant="danger" title="Post-solve hints">

Once a challenge has been solved, no additional hints should be released for that challenge. Doing so would be unfair to teams that solved it without assistance.

</Callout>
<Callout variant="tip" title="Timing considerations">

When releasing hints, consider that participating teams are often distributed across multiple time zones. Releasing a hint during one region's sleeping hours may provide an unfair advantage to teams in other regions. If possible, time hint releases to coincide with periods when most participants are likely to be active.

</Callout>
<Callout variant="note" title="Beginner-friendly challenges">

For challenges designated as beginner-friendly, it is generally acceptable to provide hints or guidance to novice teams who request help. This can support their learning and encourage participation, as long as it does not impact the final competition rankings.

</Callout>
## Announcements

Maintain clear and timely communication with participants throughout the competition. The following events should always be announced:

- Competition start and end reminders
- Challenge releases (if challenges are released in waves)
- Challenge fixes or updates
- Delays or extensions of the competition
- Hint releases
- Infrastructure issues and resolutions
- Survey release
- Writeups release

Keep announcements concise and informative. For temporal announcements, consider using Discord's [timestamp](https://r.3v.fi/discord-timestamps/) for relative times.
