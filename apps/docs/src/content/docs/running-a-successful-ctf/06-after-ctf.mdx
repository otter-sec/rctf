Following the conclusion of the competition, several administrative tasks remain. This section covers scoreboard submission to CTFtime, prize distribution procedures, feedback collection, and infrastructure teardown.

## Uploading scoreboard to CTFtime

Submitting the final scoreboard to CTFtime enables participating teams to receive ranking points. The following procedure exports the scoreboard in the format CTFtime expects.

<Steps>
<Step title="Update admin permissions">

The admin user must have the appropriate permissions to access the CTFtime integration endpoint. Update the permissions as follows:

```bash title="Terminal"
docker exec -it rctf-postgres-1 bash
```

```bash title="Inside the postgres container"
psql -U rctf
```

```sql title="Inside the postgres shell"
UPDATE users SET perms = 7 WHERE email = '[...]';
```

</Step>
<Step title="Retrieve authentication token">

Log in to the CTF platform as the admin account and execute the following snippet in the browser's developer console:

```js title="Browser console"
localStorage.getItem('token')
```

Copy the returned authentication token, removing the surrounding quotation marks.

</Step>
<Step title="Export the scoreboard">

Use the following command to retrieve the scoreboard export, replacing `TOKEN` with the authentication token from the previous step:

```bash title="Terminal"
curl https://ctf.example.com/api/v1/integrations/ctftime/leaderboard \
  -H "Authorization: Bearer TOKEN"
```

</Step>
<Step title="Submit to CTFtime">

Submit the command output through the CTFtime event dashboard.

<Callout variant="note">

Point distribution may take approximately one hour for established events. For first-time events, points are awarded only after the one-week weight voting period concludes.

</Callout>

</Step>
</Steps>
## Prize distribution

For competitions with prizes, team contact information can be retrieved directly from the database. This approach is more reliable than accepting claims via Discord, as team membership cannot be externally verified.

```bash title="Terminal"
docker exec -it rctf-postgres-1 bash
```

```bash title="Inside the postgres container"
psql -U rctf
```

```sql title="Inside the postgres shell"
SELECT * FROM users WHERE name IN ('Team 1', 'Team 2', 'Team 3');
```

<Callout variant="tip">

For competitions using divisions, the query can be extended to filter by division if prize distribution varies by category.

</Callout>
## Feedback collection

Releasing a post-competition feedback form (e.g., Google Forms) provides valuable insights for future events. Consider soliciting feedback on the following areas:

| Category       | Topics to address                                                    |
| -------------- | -------------------------------------------------------------------- |
| Infrastructure | Platform stability, challenge availability, response times           |
| Challenges     | Difficulty balance, category distribution, quality of problem design |
| Organization   | Communication clarity, support responsiveness, overall experience    |

<Callout variant="tip">

Feedback collected from participants is invaluable for improving subsequent events. Review responses thoroughly before planning the next competition.

</Callout>
## Writeup collection

Encouraging participants to publish writeups benefits the broader CTF community and provides learning resources for future competitors. Consider the following approaches:

- Announce a deadline for writeup submissions (typically one to two weeks after the event).
- Offer incentives such as prizes for outstanding writeups or recognition in official channels.
- Compile and share links to community writeups in the Discord server or on the event website.
- Release official writeups for challenges that received few or no community solutions.

## Infrastructure teardown

The timing of infrastructure teardown depends on several factors: remaining budget, the desire to keep challenges accessible for writeup authors, and the willingness to maintain operational services. Most organizers scale down infrastructure shortly after the competition concludes while maintaining minimal services for several days to allow participants to verify their solutions.

For infrastructure sponsored by cloud providers (e.g., Google Cloud credits), services may remain operational until credits are depleted, provided no sensitive data is at risk.

<Callout variant="warning" title="Teardown checklist">

Ensure the following resources are properly decommissioned to avoid unexpected costs:

- Take down the CTF platform, and back up the database and configuration before deletion to simplify setup for future events.
- Remove kCTF clusters, VPS instances, Klodd/tiny-instancer deployments, and associated GKE resources.
- For cloud services, delete static IP addresses, DNS records, load balancers, and any remaining storage buckets or artifacts.
- Rotate or revoke any API keys, tokens, or credentials used during the event.

</Callout>
