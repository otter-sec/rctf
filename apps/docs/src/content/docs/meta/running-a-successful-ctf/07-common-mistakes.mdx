These are recurring issues from CTF retrospectives and event operations. The goal is not to make every organizer paranoid, but to make the important failure modes visible early enough to handle them calmly.

## Non-technical

### Not planning properly or not having deadlines

Most event problems become harder to fix when there are no internal deadlines. Challenges can land the night before, infrastructure can still be getting provisioned an hour before opening, solution scripts can be unfinished at start time, and during-event ownership can remain unclear.

Set a few internal dates and treat them as part of the event infrastructure.

1. Stop accepting substantial challenge changes before the event. Small wording fixes and confirmed bug fixes can still happen, but new ideas should not be landing during the final preparation window.

2. Freeze infrastructure, and give production configuration enough time to run without churn. Last-minute DNS, proxy, provider, or cluster changes should be treated as risky.

3. Run a dry-run. Test registration, login, challenge visibility, instance start, flag submission, score updates, and admin triage before participants arrive. The dry-run should use production-like paths rather than only local shortcuts.

4. Every challenge should have a during-event owner, and platform operations should have a clear escalation path. When something breaks, the team should already know who is responsible for the first response.

Deadlines also keep the infrastructure owner from becoming the last-minute coordinator for unrelated delays.

## Cloud providers

### Not requesting quota in advance

New GCP, AWS, and Azure accounts often start with low default quotas. If quota is reached during the event, autoscaling stops and new instances may fail to start. Quota reviews can take multiple days, so request increases at least two weeks before the event.

<Tabs items={['GCP', 'AWS', 'Azure']}>
<Tab>

For GCP, check project-level quota in the [IAM quotas page](https://console.cloud.google.com/iam-admin/quotas). Pay attention to `CPUS`, `IN_USE_ADDRESSES`, `PERSISTENT_DISK_SSD_GB`, GPU quotas in the selected region, and the per-region GKE `SUBNETWORKS` cap.

Quota belongs to the project. Sponsored credits do not automatically raise the project limits. File the request with the event date, expected team count, expected concurrent workload, and selected region, then verify the approved limit in the same UI.

</Tab>
<Tab>

For AWS, Service Quotas requests are tied to the account and region pair. Request quota in every region you plan to use, especially for vCPU limits by instance family, Elastic IPs, and EBS volume capacity.

Region choice matters. Some instance families are easier to approve in one region than another, so choose the region only after checking whether the required capacity is realistic there.

</Tab>
<Tab>

For Azure, check regional vCPU limits by family before committing to a region. As with the other providers, credit or sponsorship does not necessarily imply the compute quota needed for an event.

Request more than the steady-state estimate. Burst, retries, node replacement, and pre-pulled images can consume more capacity than the number suggested by a simple team count.

</Tab>
</Tabs>

<Callout variant='warning' title='Request headroom'>

Ask for roughly 1.5 to 2x the worst-case estimate. The margin covers burst, retries, pre-pulled images, and replacement capacity during node churn.

</Callout>

<Callout variant='important' title='GCP trial accounts usually remain capped'>

If a GCP project is still on a free-trial billing account, quota-increase requests may be rejected regardless of timing. Trial projects are intentionally kept at lower limits.

1. Upgrade to a paid billing account before requesting quota increases. Free credits can still apply, but the billing account class matters during review.

2. Run a small always-on VM, such as an `e2-small`, for 3 to 7 days before filing the request. Consistent billable usage helps the project look like an active workload rather than a brand-new account.

3. File the request with context. Reference the active workload and the event date in the justification. A note like `currently running production workloads, scaling up for an event on YYYY-MM-DD` is clearer than a bare capacity request. Projects that move directly from no usage to a very large quota request are more likely to be denied or sent to a slower manual review queue.

</Callout>

### Exposing the instance-metadata service to challenges

Anything reachable from a challenge container may be able to reach `http://169.254.169.254` on AWS, GCP, or Hetzner, or `http://[fd00:ec2::254]` on IPv6. Those metadata services can expose short-lived credentials for the node's service account.

<Callout variant='important' title='Block metadata service access'>

Block egress from challenge containers to `169.254.0.0/16` and the IPv6 metadata addresses. Verify the rule from inside a test pod or challenge container, not only from the host.

For shared-remote VPS or bare-VM challenges without Kubernetes `NetworkPolicy`, drop IMDS at the host firewall or run the host without an attached service account.

</Callout>

<Callout variant='example' title='Verification check'>

From a test pod, run `$ curl http://169.254.169.254`. The request should fail. If the cluster is dual-stack, also test the IPv6 metadata address. On bare-metal clusters, this catches CNI configurations that silently ignore the intended policy.

</Callout>

<Callout variant='note' title='What rCTF handles'>

The [Kubernetes instancer](/integrations/instancer/kubernetes/#network-policies) includes a `NetworkPolicy` per instance namespace that excludes RFC1918, CGNAT, and link-local ranges by default. That covers IMDS on IPv4. The Terraform module also enables GKE `workload_metadata_concealment`. Opt into public egress with `<red>egress</red>: <green>true</green>` on the pod rather than removing the policy.

The bundled GKE Terraform module provisions an IPv4-only cluster, so the IPv6 IMDS address, `[fd00:ec2::254]`, is unreachable from challenge pods because the cluster has no IPv6. If you switch the cluster to dual-stack, the shipped `NetworkPolicy` has no IPv6 allow rule and will implicitly deny IPv6 egress. That keeps IPv6 IMDS blocked, but challenges that need IPv6 internet egress will also fail until you add an explicit IPv6 `IPBlock` with matching `fd00::/8` and `fe80::/10` exceptions.

</Callout>

## Kubernetes

### Skipping per-pod resource limits

A pod without `<red>resources.limits</red>` can use whatever the node has free. A fork bomb or memory leak in one instance can affect other instances on the same node.

<Callout variant='problem' title='Failure mode'>

The common version of this issue is a PodSpec with only `<red>requests</red>`, limits based on a single happy-path test, and no `<red>terminationGracePeriodSeconds</red>` cap. Held TCP connections can delay cleanup while the node remains under pressure.

</Callout>

<Callout variant='note' title='What rCTF exposes'>

The k8s-instancer `<red>instancerConfig.config.pods[]</red>` field is a real PodSpec. Set `<red>resources.requests/limits</red>` directly.

</Callout>

### Mounting broad service-account tokens

Every pod in a namespace can receive the `default` service-account token at `/var/run/secrets/kubernetes.io/serviceaccount/`. If that service account has broad permissions, or if the cluster over-grants `system:authenticated`, a compromised challenge can become Kubernetes API access.

<Callout variant='important' title='Disable tokens unless needed'>

Set `<red>automountServiceAccountToken</red>` to <green>false</green> unless the challenge genuinely needs the in-cluster API. Audit `system:authenticated` cluster-role bindings with `kubectl get clusterrolebindings -o yaml | grep -A4 system:authenticated`, and remove bindings with meaningful verbs. Also check that temporary controller RBAC changes from debugging were reverted before the event.

</Callout>

<Callout variant='note' title='What rCTF exposes'>

The k8s-instancer `<red>instancerConfig.config.pods[]</red>` field accepts the standard PodSpec, including `<red>automountServiceAccountToken</red>`.

</Callout>

## Challenges

### Shared remote without per-connection sandboxing

A shared remote where the intended solution grants code execution, such as many pwn and misc challenges, should isolate each connection. Without per-connection sandboxing, one solver can alter the binary, write a backdoor to `/tmp/`, exhaust process or file-descriptor tables, or kill the shared process. Later connections then interact with corrupted state.

<Callout variant='problem' title='Failure mode'>

The risky pattern is a pwn or RCE challenge served as one long-lived process. Each TCP connection should fork into its own sandbox with a read-only root filesystem, no network, dropped capabilities, and CPU, memory, and time limits. Also check customized kCTF templates where `nsjail` was removed, and per-team pods that still serve multiple parallel connections.

</Callout>

<Callout variant='tip' title='Recommended patterns'>

[pwn.red/jail](https://github.com/redpwn/jail) is a small base image that forks per connection inside nsjail with reasonable defaults, and it usually takes only a couple of Dockerfile lines to adopt. kCTF challenges already include nsjail when created with `kctf chal create{:sh}`. For custom services, raw [nsjail](https://github.com/google/nsjail) is still a good fit when `--max_cpus`, `--rlimit_as`, `--time_limit`, and `--disable_proc` are tuned for the challenge.

</Callout>

See the [shared-remote callout in Deploying challenges](/meta/running-a-successful-ctf/04-deployment/#shared-remote) for the deployment walkthrough.

### Distributing malware samples or forensics artifacts unprotected

Forensics, reverse engineering, and malware-analysis challenges sometimes distribute binaries that antivirus vendors will flag quickly. If a file is flagged at the bucket or CDN layer, participants may have to bypass several browser and endpoint warnings before they can download the challenge.

The highest-risk version is a sample shipped as a plain `.exe`, `.dll`, or unencrypted archive. Endpoint products across participating teams can report the file within minutes, and a CDN or storage provider may pull or blacklist it.

<Callout variant='warning' title='Archive malicious samples'>

Ship suspicious binaries inside a password-protected archive, and put the password in the challenge description. Use a non-dictionary password rather than common values like `infected`, `malware`, `password`, or the event year, since AV engines, SmartScreen, Defender, and CrowdStrike may try those values automatically.

</Callout>

### Missing runtime abuse limits

CPU and memory limits are not enough on their own. A participant can run a fork bomb like `:(){ :|:& };:`, fill `/tmp/` with `dd if=/dev/zero of=/tmp/x`, or open a large number of file descriptors from inside a challenge process. Those behaviors need runtime limits rather than assumptions about participant behavior.

<Tabs items={['Docker', 'Kubernetes', 'nsjail']}>
<Tab>

For Docker services, set `pids_limit` so fork bombs cannot exhaust the host `kernel.pid_max`. Set `ulimits.nofile` so a process cannot exhaust the file-descriptor table. If `/tmp/` is writable, use a size-limited `tmpfs`. Bare-metal challenges need the same storage cap with a `tmpfs` `size=` option.

</Tab>
<Tab>

For Kubernetes services, set `resources.limits.ephemeral-storage` on pods and configure kubelet `podPidsLimit` at the cluster level. Use `<red>readOnlyRootFilesystem</red>` where practical and set `<red>terminationGracePeriodSeconds</red>` so cleanup cannot be delayed indefinitely.

</Tab>
<Tab>

For nsjail services, use `--rlimit_nproc`, `--rlimit_nofile`, and `--time_limit`. Shared remote services should have a per-connection time limit so slow or long-lived TCP connections cannot starve the connection table.

</Tab>
</Tabs>

<Callout variant='example' title='Abuse test'>

Before release, test challenge services with intentionally abusive clients. Include `:(){ :|:& };:`, `dd if=/dev/zero of=/tmp/x`, `for i in $(seq 100000); do exec {fd}<>/dev/null; done`, and slow TCP connections where the challenge uses a shared remote.

</Callout>

<Callout variant='note' title='What rCTF handles'>

The [Docker instancer](/integrations/instancer/docker/#docker-challenge-config) defaults to `6m` memory, `1.0` CPU, `1024` PIDs, and `1024` nofile per service when not overridden. It also exposes `<red>mem_limit</red>`, `<red>cpus</red>`, `<red>pids_limit</red>`, and `<red>ulimits</red>` for tuning. The GKE Terraform module sets kubelet `<red>podPidsLimit</red>` to `1024{:ts}` cluster-wide through `<red>gcp_instancer_pod_pids_limit</red>`, so a fork bomb inside a challenge pod hits that cap before the node `kernel.pid_max`.

The Kubernetes side otherwise leaves several settings to the challenge author. Ephemeral-storage limits, `<red>readOnlyRootFilesystem</red>`, FD ulimits, and `<red>terminationGracePeriodSeconds</red>` are not defaulted. See the [per-pod safety checklist](/integrations/instancer/kubernetes/#per-pod-safety-checklist) on the Kubernetes instancer page for the full list of fields to set.

</Callout>

<Callout variant="tip">
  When in doubt, treat every challenge image as already compromised. Assume participants can get
  code execution inside the challenge container, even when the intended solution is not RCE.
</Callout>

## Miscellaneous

### Improper host hardening

A cloud VM with an unpatched kernel, exposed Docker socket, or unnecessary privileges can turn challenge code execution into host access. Container-escape and local privilege-escalation issues are patched regularly, but unpatched hosts remain exposed.

<Callout variant='important' title='Keep the host boundary intact'>

Run unattended upgrades or an equivalent process on challenge hosts, then reboot into the newer kernel before the event. Do not mount `/var/run/docker.sock` into challenge containers. Avoid `--privileged`, `--cap-add=SYS_ADMIN`, `--pid=host`, and `--net=host` unless the security impact has been reviewed. Keep runc, Docker, and containerd on patched versions.

Public local privilege-escalation exploits can appear within days of disclosure. A container with Docker socket access can also start a container with `docker run -v /:/host` and walk out to the host filesystem. Pin minimum runtime versions and verify them before the event.

</Callout>

### Challenge machine runs out of disk space

Disk exhaustion is easy to miss because CPU and memory dashboards can remain quiet while logs, images, and caches grow in the background.

<Callout variant='warning' title='Common disk sources'>

Docker `json-file` logs have no rotation unless `max-size` and `max-file` are set in `/etc/docker/daemon.json`, so a noisy challenge can fill `/var/lib/docker/containers/*/.log` for the lifetime of the container. `journald` can grow in the same way unless `SystemMaxUse=` is configured. Rebuilds and redeploys also leave dangling images and build cache in `/var/lib/docker/`, so schedule `docker system prune` or monitor disk usage closely before the event.

</Callout>

### Client-IP extraction not tested end-to-end

Every layer in front of the platform, such as Cloudflare, an L7 load balancer, nginx, or Traefik, can rewrite the source address. If the chain is not configured correctly, the platform and challenges may see the proxy IP rather than the participant IP. The failure is often silent until rate limits or bans behave incorrectly. A bad chain can ban a shared Cloudflare edge IP and lock participants out, or make every request appear to come from `127.0.0.1`. The challenge-side version is also important, since a web challenge that trusts `X-Forwarded-For` without validating the upstream hop can be spoofed by participants.

<Callout variant='example' title='Production proxy test'>

Send a request through the real DNS from a known external IP and confirm the admin or audit log shows that IP. Then test a challenge through the same path and confirm the challenge sees the same address. A staging path without the full proxy chain is not enough for this check.

</Callout>

<Callout variant='important' title='Trust only the edge you control'>

If `USE_PROXY_HEADERS=true{:ts}` is enabled without a locked-down trusted-proxy CIDR, participants can spoof `X-Forwarded-For`. The same applies to nginx `set_real_ip_from` and Traefik `forwardedHeaders.trustedIPs` when the trusted range is wider than the proxy layer you control. With Cloudflare, restore `CF-Connecting-IP` so the platform does not log only Cloudflare edge IPs.

</Callout>

### Not being prepared for traffic

The first few minutes after start combine concurrent registrations, logins, challenge reads, and leaderboard reads. Load-test those paths from a separate machine before the event, and size the challenge cluster for `expected teams × overlapping instanced challenges`.

<Callout variant='warning' title='Capacity checks'>

Enable captcha for sensitive unauthenticated actions, verify rate limits on sensitive platform paths, and configure trusted proxy IP ranges before the load test. The test should use the production proxy chain rather than a local or staging shortcut.

</Callout>

<Callout variant='note' title='What rCTF handles'>

rCTF helps with this through the leaderboard worker and Redis cache, nginx brotli and gzip support, immutable cache headers, [Docker](/integrations/instancer/docker/) and [Kubernetes](/integrations/instancer/kubernetes/) instancers for per-team lifecycles, GKE module autoscaling, captcha integrations for [reCAPTCHA, hCaptcha, and Turnstile](/providers/captcha/), and default route rate limits.

</Callout>

## Onsite

### Connection issues

The venue network is often different from the network used during testing. Rented venues, university lecture halls, and hotel conference rooms may use eduroam, vendor-managed Wi-Fi, captive portals, or enterprise firewalls from vendors such as Fortinet, Palo Alto, or Cisco. Discovering those restrictions on the day of the event leaves little time to adapt.

<Tabs items={['Wireless', 'Firewall', 'NAT']}>
<Tab>

Ask for the per-AP and per-room bandwidth cap, then compare it against the expected laptop count. A rough warning sign is one access point expected to serve around 10 teams, or 40 laptops pulling challenge files, Docker images, or VPN updates at once.

</Tab>
<Tab>

Ask for the firewall vendor, blocked ports, and whether a temporary exception can be added. Some managed networks block outbound SSH on TCP port `22`, arbitrary high or low ports, or traffic that resembles tunneling. DPI appliances can pass HTTPS while dropping unrecognized TCP payloads. Test actual challenge solutions end to end on the venue network.

</Tab>
<Tab>

NAT or CGNAT can collapse many teams to one source IP. Any IP-keyed rate limit can then affect legitimate traffic. Whitelist the venue egress range or relax the relevant limits for onsite traffic.

</Tab>
</Tabs>

<Callout variant='tip'>

If the venue cannot provide network details, plan a backup uplink and test it before participants arrive. Ask for the egress firewall vendor, blocked ports, per-AP and per-room bandwidth caps, and whether temporary firewall exceptions are available.

</Callout>

### Not enough VPN configs

Onsite CTFs that also allow remote participation sometimes provide WireGuard or OpenVPN configs so challenge networks are reachable without exposing all services publicly. Two recurring issues are generating too few configs and binding the VPN only to the venue LAN. A common failure is having enough configs for a few onsite devices, but none for remote players. Either issue can force remote teams to build their own access path, which adds friction and can create an uneven experience.

<Callout variant='warning' title='Prepare the access pool early'>

Generate configs for all players, not only the people attending onsite. Pre-generating a large pool, such as 100 configs, is inexpensive and avoids last-minute manual work. Bind the VPN endpoint to a public IP, or use a cloud VPS that bridges to the venue network, so remote players can reach it over the public internet.

</Callout>
