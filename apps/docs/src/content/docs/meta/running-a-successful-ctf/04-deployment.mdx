With the CTF platform operational and challenges ready to be created, the next step is deployment. There are two primary deployment types alongside static challenges: **shared remote** and **instanced remote**.

<Callout variant="tip">

Regardless of deployment method, it is considered good practice to include a local setup of the challenge for debugging purposes (e.g., using Docker). This also simplifies remote deployment, as most solutions are designed around running a provided Docker image.

</Callout>
Additionally, rCTF supports hosting admin bots separately from the challenge to handle traffic spikes and provide a more accurate simulation of external visitors, rather than relying on internal IPs as would occur in a local deployment.

## Shared remote

In a shared remote, every competitor gets access to a single shared instance between all competitors. This is convenient when the vulnerability does not depend on state or is sandboxed, for example, cryptographic vulnerability that proves you're able to recover the secret value from the server.

Google has created [kCTF](https://google.github.io/kctf/introduction.html) for this purpose, which enables challenge deployment on GCP with instance scaling, health checks, automatic restarts upon failure, and secure sandboxing on shared hosts. For a simpler alternative, shared instances can also be run on a traditional VPS either inside Docker or directly on the host.

<Callout variant='warning' title='Wrap shared remotes in a jail'>
  Any shared remote where the intended solve gives a competitor code execution must run the
  vulnerable process inside a per-connection sandbox. Concretely:

- **[pwn.red/jail](https://github.com/redpwn/jail)** is the modern drop-in - a small base image that forks the challenge per connection inside an nsjail with read-only rootfs, no network, dropped caps, and CPU/memory/time limits. Add two lines to a Dockerfile (`FROM pwn.red/jail`) and you get isolation per session for free. Recommended default for pwn / RCE-style shared remotes.
- **[nsjail](https://github.com/google/nsjail)** is the lower-level building block both kCTF and pwn.red/jail wrap. kCTF runs every challenge inside an nsjail by default; if you're not on kCTF and not using pwn.red/jail, configure nsjail yourself.

Use one of these everywhere it's possible. A shared remote without per-connection isolation lets the first solver replace the binary, drop a backdoor in `/tmp/{:dir}`, or just kill the process for everyone else.

</Callout>

### kCTF

kCTF maintains its own [documentation](https://google.github.io/kctf/), so this section focuses on considerations that may not be immediately apparent, particularly for first-time users:

- When deploying kCTF, you have the option to control under which (sub)domain the challenges are available. The `kctf cluster create{:sh}` command has the `--domain-name` option responsible for this, and you should expect to either use nameservers of Google if you want to host challenges under `*.example.com`, or use a subdomain with a `NS` records pointing to Google so that challenges are available under `*.subdomain.example.com`.
- By default, the created nodes are spot (preemptible) instances. This means that every 24 hours, the nodes will be rotated, causing temporary downtime for the challenges. To avoid this behavior, trigger a cluster resize without the spot instance flag (e.g., `kctf cluster resize --min-nodes 1 --max-nodes 3 --num-nodes 1 --machine-type n2-standard-4{:sh}`), which incurs slightly higher costs.
- If a health check fails, the challenge will be automatically restarted, which may manifest as apparent connectivity issues. In such scenarios, verify the output of `kctf chal status{:sh}` to expedite debugging.
- Monitor your GCP project's quotas, as the dashboard indicates which limits have been reached or are approaching capacity. Deploying a large number of challenges may cause deployment failures due to insufficient IP addresses or other quota constraints. Configuring kCTF in advance serves as a practical method to verify that quotas are sufficient and may improve your account's standing for automatic quota increase approvals.
- Some challenges deployed to kCTF may require additional modifications due to security hardening features. Configuring kCTF in advance is highly recommended to identify and resolve such issues before the competition begins. The [Kubernetes Pods documentation](https://kubernetes.io/docs/concepts/workloads/pods/) provides additional context on configurable options for kCTF challenge instances (for example, additional volumes, as the file system is read-only by default).

### Docker

If the complexity of kCTF is undesirable due to GCP and Kubernetes requirements, a simpler option is to deploy challenges using Docker on one or more dedicated hosts. It is advisable to avoid reusing the same VPS as the CTF platform, as challenges may receive significant traffic volumes and isolating participant data from intentionally vulnerable services is a recommended security practice.

The most straightforward deployment method is [Docker Compose](https://docs.docker.com/compose/), with each challenge assigned a unique port. Note that challenges resulting in remote code execution carry risks of container breakouts (particularly if elevated privileges are required) or malicious actions such as disk exhaustion. Wrap the challenge process in [pwn.red/jail](https://github.com/redpwn/jail) or nsjail - see the [shared-remote callout above](#shared-remote).

<Callout variant="danger">

It is extremely important to make sure competitors are not able to access any internal metadata services (e.g. `http://169.254.169.254` on AWS/GCP cloud providers which have access credentials to the cloud account).

</Callout>
### VM

An alternative approach is to provision a separate VM (or VPS) for each challenge instance. Although more costly than running all challenges on a single host, this configuration reduces the blast radius, preventing a single compromised challenge from affecting others through disk exhaustion, container breakouts, or similar attacks. The optimal configuration depends on the specific requirements of each challenge.

<Callout variant="danger">

It is extremely important to make sure competitors are not able to access any internal metadata services (e.g. `http://169.254.169.254` on AWS/GCP cloud providers which have access credentials to the cloud account).

</Callout>
## Instanced remote

In an instanced remote configuration, each competitor receives a dedicated instance. This is necessary for challenges where the vulnerability inherently provides file write or remote code execution capabilities, which could otherwise be exploited by competitors to disrupt challenge functionality, remove flags, or interfere with other participants.

rCTF v2 ships first-party support for instanced challenges, so you don't need a third-party integration here. Pick whichever backend matches your operational comfort level:

- **[Docker instancer](/integrations/instancer/docker/)** - bundled Python FastAPI service that runs on a standalone Docker host alongside Traefik and Redis. Lightweight; good fit for small-to-medium events.
- **[Kubernetes instancer](/integrations/instancer/kubernetes/)** - Go operator that reconciles `ChallengeInstance` custom resources into per-team namespaces with network policies, Traefik routes, and ACME wildcard TLS. Comes with a Terraform module for GKE. The scalable option.

The shared `<red>instancerConfig</red>` schema, participant lifecycle, and admin UI are documented in the [instancer overview](/integrations/instancer/).

If you'd rather stick with an existing setup, third-party integrations like [Klodd](https://klodd.tjcsec.club/) still work against rCTF, but the first-party instancer is the recommended path going forward.

<Callout variant="warning">

Instances can be deployed before the CTF begins. Exercise caution when deploying challenges in advance if their names are predictable.

</Callout>

## Admin bot

rCTF v2 ships a first-party admin bot integration that runs trusted TypeScript handlers in a Puppeteer-driven Chrome/Firefox session. The handlers live alongside the challenge, validate participant input, and stream structured per-job logs back to the platform. Configuration, the handler API, scaling notes, and deployment guidance are all in the dedicated [Admin bot](/integrations/admin-bot/) page.

## Keeping challenges in sync

Updating a challenge is where most operational pain comes from. Touching the source code without updating the published attachment leaves teams solving against stale handouts; rebuilding the Docker image but forgetting to roll the running deployment leaves the flag pointing at the old binary; tweaking `<red>instancerConfig</red>` in the admin UI without bumping the image causes new instances to come up with mismatched assets. Each of these is easy to miss in the moment and hard to debug after the fact.

<Callout variant="tip" title="Use Konata + CI as the single source of truth">
  Commit every challenge's `kona.yml{:file}` alongside the source, and let [Konata](/integrations/konata/)
  drive deployment from CI on every push to `main`. With the [Konata GitHub
  Action](/integrations/konata/#ci-integration), one commit can rebuild and push the Docker image,
  re-render attachments, refresh metadata on rCTF, and (for k8s-instancer) trigger a rollout of the
  running deployment - all from the same source tree, in the right order.
</Callout>

Pair that with **generated attachments** rather than hand-copied files. If a challenge produces its handout from the build (a multi-stage `Dockerfile{:file}` export, a `make handout` target, a script), reference that path from `<red>attachments.files</red>` and let Konata's [deterministic compressor](/integrations/konata/#kona-compress) re-archive it on every sync. Same inputs always produce the same archive bytes, so rCTF's content-hash dedup leaves attachments alone when nothing changed and uploads exactly once when something did. You stop shuttling tarballs into the right folder, and the published attachment is always the one that came out of the latest build.

The DiceCTF Quals 2026 challenge repository is a working reference for this workflow end-to-end - challenge sources, `kona.yml{:file}` files, and a [`deploy.yaml{:file}`](https://github.com/dicegang/dicectf-quals-2026-challenges/blob/main/.github/workflows/deploy.yaml) that ships every change through Konata.
