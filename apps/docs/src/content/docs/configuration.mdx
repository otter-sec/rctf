rCTF is configured through YAML or JSON files in a `rctf.d/` directory and optional environment variable overrides.

## Configuration loading

Configuration is loaded from a directory named `rctf.d/`. The loader searches upward from the `packages/config/` directory, or you can specify a path via the `RCTF_CONF_PATH{:sh}` environment variable.

All files in the directory (`.yaml`, `.yml`, or `.json`) are loaded **alphabetically** and deep-merged in order. This allows you to split configuration across multiple files for organization:

<FileTree>
  <Dir name='rctf.d'>
    <File name='01-base.yaml' comment='Core settings' />
    <File name='02-providers.yaml' comment='Provider configuration' />
    <File name='03-overrides.yaml' comment='Environment-specific overrides' />
  </Dir>
</FileTree>

Environment variables are applied last, overriding any file-based values.

## Environment variables

The following environment variables are supported. They override values from config files.

### Core

| Variable                  | Type          | Description                                              |
| ------------------------- | ------------- | -------------------------------------------------------- |
| `RCTF_NAME{:sh}`          | `string{:ts}` | CTF display name                                         |
| `RCTF_ORIGIN{:sh}`        | `string{:ts}` | CTF origin URL (e.g., `https://ctf.example.com`)         |
| `RCTF_TOKEN_KEY{:sh}`     | `string{:ts}` | Base64-encoded 32-byte key for token encryption          |
| `RCTF_INSTANCE_TYPE{:sh}` | `string{:ts}` | `[32mall[0m{:ansi}`, `[32mfrontend[0m{:ansi}`, or `[32mleaderboard[0m{:ansi}` |
| `RCTF_CONF_PATH{:sh}`     | `string{:ts}` | Path to config directory (overrides search)              |

### Database

| ----------------------------- | -------------- | ------------------------------------------------- |

| `RCTF_DATABASE_PORT{:sh}`     | `integer{:ts}` | PostgreSQL port                                   |
| `RCTF_DATABASE_USERNAME{:sh}` | `string{:ts}`  | PostgreSQL user                                   |
| `RCTF_DATABASE_PASSWORD{:sh}` | `string{:ts}`  | PostgreSQL password                               |
| `RCTF_DATABASE_DATABASE{:sh}` | `string{:ts}`  | PostgreSQL database name                          |
| `RCTF_DATABASE_MIGRATE{:sh}`  | `string{:ts}`  | `[32mbefore[0m{:ansi}`, `[32monly[0m{:ansi}`, or `[32mnever[0m{:ansi}` |

## Configuration reference

Below is the complete reference of all configuration options available in `rctf.d/` files.

### Core settings

```yaml
ctfName: My CTF # CTF display name (required)
origin: https://ctf.example.com # Public origin URL (required)
tokenKey: <base64-32-byte-key> # Token encryption key (required)
instanceType: all # all | frontend | leaderboard
```

| Field                 | Type          | Default      | Description                                                                                                                              |
| --------------------- | ------------- | ------------ | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `[31mctfName[0m{:ansi}`      | `string{:ts}` | -            | Display name of your CTF                                                                                                                 |
| `[31morigin[0m{:ansi}`       | `string{:ts}` | -            | Public URL of your CTF (no trailing slash)                                                                                               |
| `[31mtokenKey[0m{:ansi}`     | `string{:ts}` | -            | Base64-encoded 32-byte key for AES-GCM token encryption                                                                                  |
| `[31minstanceType[0m{:ansi}` | `string{:ts}` | `[32mall[0m{:ansi}` | Controls which components run: `[32mall[0m{:ansi}` (API + leaderboard worker), `[32mfrontend[0m{:ansi}` (API only), `[32mleaderboard[0m{:ansi}` (worker only) |

<Callout variant="warning">

The `[31mtokenKey[0m{:ansi}` is critical for security. All authentication tokens are encrypted with this key. If it is lost or changed, all existing tokens become invalid and users will need to re-authenticate.

</Callout>
### Database

```yaml
database:
  sql: postgres://user:password@localhost:5432/rctf
  redis: redis://localhost:6379
  migrate: never
```

The `[31msql[0m{:ansi}` field accepts either a connection string or an object:

```yaml
database:
  sql:
    host: localhost
    port: 5432
    user: rctf
    password: secret
    database: rctf
    maxPoolSize: 100 # Default: 100
    idleTimeout: 30000 # Default: 30000 (ms)
    connectTimeout: 3000 # Default: 3000 (ms)
```

The `[31mredis[0m{:ansi}` field accepts either a connection string or an object:

```yaml
database:
  redis:
    host: localhost
    port: 6379
    password: secret # Optional
    database: 0 # Optional, default 0
```

| Field                     | Type                           | Default        | Description                                                                                               |
| ------------------------- | ------------------------------ | -------------- | --------------------------------------------------------------------------------------------------------- |
| `[31mdatabase.sql[0m{:ansi}`     | `string{:ts}` \| `object{:ts}` | -              | PostgreSQL connection (required)                                                                          |
| `[31mdatabase.redis[0m{:ansi}`   | `string{:ts}` \| `object{:ts}` | -              | Redis connection (required)                                                                               |
| `[31mdatabase.migrate[0m{:ansi}` | `string{:ts}`                  | `[32mnever[0m{:ansi}` | `[32mbefore[0m{:ansi}` runs migrations on startup, `[32monly[0m{:ansi}` runs migrations and exits, `[32mnever[0m{:ansi}` skips |

### Timing

```yaml
startTime: 1735689600000 # January 1, 2025 00:00 UTC
endTime: 1735776000000 # January 2, 2025 00:00 UTC
```

| Field              | Type          | Default | Description                                            |
| ------------------ | ------------- | ------- | ------------------------------------------------------ |
| `[31mstartTime[0m{:ansi}` | `number{:ts}` | -       | Competition start time in Unix milliseconds (required) |
| `[31mendTime[0m{:ansi}`   | `number{:ts}` | -       | Competition end time in Unix milliseconds (required)   |

<Callout variant="tip">

To convert a date to Unix milliseconds: `date -d "2025-01-01T00:00:00Z" +%s000{:sh}` or use `new Date('2025-01-01T00:00:00Z').getTime()` in JavaScript.

</Callout>
### Divisions

```yaml
divisions:
  open: Open
  student: Student
defaultDivision: open
divisionACLs:
  - match: domain
    value: edu
    divisions: [student, open]
  - match: any
    value: ''
    divisions: [open]
```

| Field                    | Type          | Default            | Description                                   |
| ------------------------ | ------------- | ------------------ | --------------------------------------------- |
| `[31mdivisions[0m{:ansi}`       | `object{:ts}` | `{ open: "Open" }` | Map of division ID to display name            |
| `[31mdefaultDivision[0m{:ansi}` | `string{:ts}` | -                  | Default division for new users (optional)     |
| `[31mdivisionACLs[0m{:ansi}`    | `array{:ts}`  | -                  | Access control rules for divisions (optional) |

#### Division ACLs

ACLs control which divisions a user can register for based on their email. Each ACL entry has:

| Field              | Description                                                                                   |
| ------------------ | --------------------------------------------------------------------------------------------- |
| `[31mmatch[0m{:ansi}`     | Match type: `[32mdomain[0m{:ansi}`, `[32memail[0m{:ansi}`, `[32mregex[0m{:ansi}`, or `[32many[0m{:ansi}`                  |
| `[31mvalue[0m{:ansi}`     | Value to match against (domain suffix, exact email, regex pattern, or empty for `[32many[0m{:ansi}`) |
| `[31mdivisions[0m{:ansi}` | Array of division IDs this rule grants access to                                              |

ACL rules are evaluated in order. The first matching rule determines the user's allowed divisions.

<Callout variant="note">

Division ACLs only apply when an email provider is configured. If CTFtime authentication is enabled, ACLs are bypassed entirely - all divisions become available.

</Callout>
### Authentication

```yaml
registrationsEnabled: true
userMembers: true
maxMembers: 50
loginTimeout: 3600000
ctftime:
  clientId: '12345'
  clientSecret: your-secret
```

| Field                         | Type           | Default        | Description                                                |
| ----------------------------- | -------------- | -------------- | ---------------------------------------------------------- |
| `[31mregistrationsEnabled[0m{:ansi}` | `boolean{:ts}` | `true{:ts}`    | Whether new registrations are allowed                      |
| `[31muserMembers[0m{:ansi}`          | `boolean{:ts}` | `true{:ts}`    | Enable team members feature                                |
| `[31mmaxMembers[0m{:ansi}`           | `number{:ts}`  | `50{:ts}`      | Maximum members per team                                   |
| `[31mloginTimeout[0m{:ansi}`         | `number{:ts}`  | `3600000{:ts}` | Verification/CTFtime token expiry in milliseconds (1 hour) |
| `[31mctftime.clientId[0m{:ansi}`     | `string{:ts}`  | -              | CTFtime OAuth client ID (numeric string)                   |
| `[31mctftime.clientSecret[0m{:ansi}` | `string{:ts}`  | -              | CTFtime OAuth client secret                                |

<Callout variant="note">

Auth tokens (used for logging in) never expire. Only verification and CTFtime tokens expire according to `[31mloginTimeout[0m{:ansi}`.

</Callout>
