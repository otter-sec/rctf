This guide covers setting up rCTF from source without Docker, suitable for development or environments where you need full control over the stack.

## Prerequisites

- [Bun](https://bun.sh/) 1.2 or later
- PostgreSQL 15+ with the `pg_trgm` extension enabled
- Redis 7+

## Setup

<Steps>
<Step title="Clone and install dependencies">

```ansi
$ [31mgit[0m clone https://github.com/otter-sec/rctf.git
$ [31mcd[0m rctf
$ [31mbun[0m i
```

</Step>
<Step title="Create a configuration directory">

rCTF loads configuration from a `rctf.d/` directory. Files inside it (YAML or JSON) are loaded alphabetically and deep-merged. Create one with `$ [31mmkdir[0m rctf.d{:ansi}` at the project root.

<FileTree>
  <Dir name='rctf.d'>
    <File name='01-base.yaml' highlight comment='create this next' />
  </Dir>
</FileTree>

Create a minimal configuration file:

```yaml title="rctf.d/01-base.yaml" showLineNumbers {"Generate with: $ openssl rand -base64 32":3-4} {"Unix milliseconds":11-12}
ctfName: My CTF
origin: https://ctf.example.com

tokenKey: <base64-encoded-32-byte-key>

database:
  sql: postgres://user:password@localhost:5432/rctf
  redis: redis://localhost:6379
  migrate: before

startTime: 1735689600000
endTime: 1735776000000

divisions:
  open: Open
```

<Callout variant="tip" title="Generating a token key">

    The `[31mtokenKey[0m{:ansi}` must be a base64-encoded 32-byte random key used for AES-GCM token encryption. Generate one with `$ [31mopenssl[0m rand -base64 32{:ansi}`.

</Callout>

You can also set configuration values through environment variables. See [Configuration](/configuration/) for the full reference.

</Step>
<Step title="Run database migrations">

```ansi
$ [31mbun[0m run db:migrate
```

Alternatively, set `[31mdatabase.migrate[0m{:ansi}` to `[32mbefore[0m{:ansi}` in your config to run migrations automatically on startup.

</Step>
<Step title="Start the development servers">

```ansi
$ [31mbun[0m dev
```

This starts both the API server on `http://127.0.0.1:3000` and the SvelteKit dev server on `http://127.0.0.1:5173`. The dev server proxies API requests to the backend.

To run them separately:

```ansi
$ [31mbun[0m run dev:api   [02m# API only on :3000[0m
$ [31mbun[0m run dev:web   [02m# Frontend only on :5173[0m
```

<Callout variant="tip" title="One-shot dev setup">

    For a fresh local environment, `$ [31mbun[0m run dev:mock{:ansi}` chains `$ [31mbun[0m run db:migrate{:ansi}`, `$ [31mbun[0m run dev:seed{:ansi}`, and `$ [31mbun[0m run dev{:ansi}` together. The seed resets the database and populates it with an admin account, 48 teams, 18 challenges across `rev`/`pwn`/`crypto`/`web`/`misc`, plus realistic solves and submissions. Login URLs for the admin and a sample team are printed to the console.

</Callout>

</Step>
</Steps>

## Production build

Build both the API and frontend:

```ansi
$ [31mbun[0m run [02m--filter[0m [32m'@rctf/web'[0m build   [02m# Builds static frontend[0m
$ [31mbun[0m run [02m--filter[0m [32m'@rctf/api'[0m start   [02m# Starts API server[0m
```

The API server listens on the port specified by the `PORT{:sh}` environment variable (default `3000{:ts}`). In production, the static frontend is typically served by Nginx or a similar reverse proxy.

## Scaling with instance types

The `[31minstanceType[0m{:ansi}` configuration option controls which components run in a given process:

| Value                         | Behavior                                            |
| ----------------------------- | --------------------------------------------------- |
| `[32mall[0m{:ansi}` (default) | Runs both the API server and the leaderboard worker |
| `[32mfrontend[0m{:ansi}`      | Runs the API server only                            |
| `[32mleaderboard[0m{:ansi}`   | Runs the leaderboard computation worker only        |

This allows horizontal scaling by running separate `[32mfrontend[0m{:ansi}` and `[32mleaderboard[0m{:ansi}` instances:

```yaml title="rctf.d/02-scaling.yaml"
instanceType: frontend # or 'leaderboard'
```

## Running tests

Tests use PGlite (in-process PostgreSQL) and ioredis-mock, so no external services are needed:

```ansi
$ [31mbun[0m test                    [02m# Run all tests from root[0m
$ [31mbun[0m run test:coverage       [02m# With coverage[0m
$ [31mbun[0m run typecheck           [02m# Typecheck all workspaces[0m
```
