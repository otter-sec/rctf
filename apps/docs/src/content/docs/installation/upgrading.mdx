This guide covers the changes between rCTF v1 and v2 and what you need to do to upgrade.

## Database migration

Database schema changes are handled automatically by Drizzle ORM. Run migrations before starting the new version:

```sh
$ bun run db:migrate
```

Or set `<red>database.migrate</red>` to `<green>before</green>` in your configuration to migrate on startup.

### New tables

| Table            | Purpose                                                                    |
| ---------------- | -------------------------------------------------------------------------- |
| `admin_bot_jobs` | Tracks admin bot job queue (status, inputs, logs)                          |
| `settings`       | Stores runtime-editable settings (CTF name, client timing, sponsors, etc.) |

### New columns on `users`

| Column                   | Type      | Description                          |
| ------------------------ | --------- | ------------------------------------ |
| `avatar_url`             | text      | URL to user's avatar image           |
| `country_code`           | text      | ISO 3166-1 alpha-2 country code      |
| `status_text`            | text      | User status message (max 60 chars)   |
| `score`                  | integer   | Cached current score                 |
| `global_rank`            | integer   | Cached global leaderboard rank       |
| `division_rank`          | integer   | Cached division-specific rank        |
| `last_solve_at`          | timestamp | Time of last solve (for tiebreaking) |
| `last_tiebreak_solve_at` | timestamp | Time of last tiebreak-eligible solve |

### New columns on `challenges`

| Column        | Type    | Description                    |
| ------------- | ------- | ------------------------------ |
| `score`       | integer | Cached current challenge score |
| `solve_count` | integer | Cached number of solves        |

## API changes

All v1 routes continue to work. The v2 API is available at `<black>/api/v2/...</black>` and includes several improvements and new endpoints.

### Captcha field rename

V2 routes use `<red>captchaCode</red>` instead of `<red>recaptchaCode</red>` in request bodies. This applies to all endpoints that support captcha validation.

### File uploads

The v2 admin upload endpoint (`<green>POST</green> <black>/api/v2/admin/upload</black>`) accepts binary files via `multipart/form-data` instead of base64-encoded data URIs used in v1.

### New v2 endpoints

| Endpoint | Description |
| --- | --- |
| `<cyan>GET</cyan> <black>/v2/auth/verify-info</black>` | Returns info about a verification token |
| `<cyan>GET</cyan> <black>/v2/leaderboard/challs</black>` | Challenge metadata with first 3 solvers per challenge |
| `<cyan>GET</cyan> <black>/v2/leaderboard/with-graph</black>` | Combined leaderboard and graph in one request |
| `<orange>PATCH</orange> <black>/v2/users/me/avatar</black>` | Upload and moderate avatar images |
| `<cyan>GET</cyan>/<green>POST</green> <black>/v2/admin/users</black>` | List all users with search, filters, and pagination |
| `<cyan>GET</cyan>/<magenta>PUT</magenta> <black>/v2/admin/settings</black>` | Runtime settings management |
| `<green>POST</green> <black>/v2/admin/users/:id/token</black>` | Generate login token for a team |
| `<red>DELETE</red> <black>/v2/admin/challs/:id/solves/:userId</black>` | Delete a specific solve |
| `<cyan>GET</cyan> <black>/v2/admin/instancer/schema</black>` | Instancer config JSON schema |
| `<cyan>GET</cyan>/<green>POST</green> <black>/v2/admin/admin-bot/*</black>` | Admin bot job management |
| `<cyan>GET</cyan>/<magenta>PUT</magenta>/<orange>PATCH</orange>/<red>DELETE</red> <black>/v2/integrations/challs/:id/instance</black>` | Challenge instance lifecycle |
| `<cyan>GET</cyan>/<green>POST</green> <black>/v2/integrations/challs/:id/admin-bot/*</black>` | Admin bot submission and status |

### Response differences

- Challenge files in v2 include a `size` field (v1 did not track file sizes)
- Challenge solves in v2 include user profile data (avatar, country, status) and blood status
- Leaderboard in v2 supports a `search` query parameter for team name search (rate limited)

## Configuration changes

### Captcha provider

The v1 `<red>recaptcha</red>` top-level config is automatically converted to the new `<red>captcha.provider</red>` format at startup. However, migrating to the new format is recommended:

<Tabs items={['v1 (deprecated)', 'v2 (recommended)']}>
<Tab>

```yaml title="v1"
recaptcha:
  siteKey: your-site-key
  secretKey: your-secret-key
  protectedActions:
    - register
```

</Tab>
<Tab>

```yaml title="v2"
captcha:
  provider:
    name: captcha/recaptcha
    options:
      siteKey: your-site-key
      secretKey: your-secret-key
  protectedEndpoints:
    - register
```

</Tab>
</Tabs>

The new format also supports `<green>captcha/hcaptcha</green>` and `<green>captcha/turnstile</green>` providers.

### Analytics provider

The v1 `<red>globalSiteTag</red>` config is automatically converted to the `<red>analytics.provider</red>` format. Migrate to:

```yaml title="v2 (recommended)"
analytics:
  provider:
    name: analytics/google
    options:
      siteTag: G-XXXXXXXX
```

Additional analytics providers are available: `<green>analytics/cloudflare</green>` and `<green>analytics/plausible</green>`.

### New configuration sections

The following configuration sections are new in v2:

| Section                        | Purpose                                      |
| ------------------------------ | -------------------------------------------- |
| `<red>adminBot</red>`          | Admin bot provider and settings              |
| `<red>avatarsModeration</red>` | Avatar content moderation (OpenAI)           |
| `<red>bloodBot</red>`          | First blood notifications (Discord/Telegram) |
| `<red>analytics</red>`         | Analytics provider                           |
| `<red>proxy</red>`             | Reverse proxy and Cloudflare settings        |
| `<red>maxAvatarSize</red>`     | Maximum avatar upload size (default 1 MB)    |

See the [Configuration](/configuration/) reference for full details on all options.
