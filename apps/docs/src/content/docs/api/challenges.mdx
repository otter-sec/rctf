Challenge routes expose public challenge metadata, solve history, and the legacy flag submission endpoint. Admin challenge mutation routes are documented in [Admin](/api/admin/).

## Routes

| Method | Path | Version | Auth | Gate |
| --- | --- | --- | --- | --- |
| `<route>GET</route>` | `<route>/api/v2/challs</route>` | V2 | Public + optional bypass (`challsRead{:ts}`) | Started |
| `<route>GET</route>` | `<route>/api/v2/challs/:id/solves</route>` | V2 | Optional + bypass (`challsRead{:ts}`) | Started |
| `<route>GET</route>` | `<route>/api/v1/challs</route>` | V1 | Public + optional bypass (`challsRead{:ts}`) | Started |
| `<route>GET</route>` | `<route>/api/v1/challs/:id/solves</route>` | V1 | Public + optional bypass (`challsRead{:ts}`) | Started |
| `<route>POST</route>` | `<route>/api/v1/challs/:id/submit</route>` | V1 | Required | Started + not ended |

## V2 routes

### `<route>GET /api/v2/challs</route>`

Returns the visible challenge list with V2 integration metadata.

| Property       | Value                                        |
| -------------- | -------------------------------------------- |
| Auth           | Public + optional bypass (`challsRead{:ts}`) |
| Gate           | Started                                      |
| Permissions    | -                                            |
| Captcha action | -                                            |
| Rate limit     | -                                            |

**Response**

`<green>200 goodChallenges</green>` returns an array of challenge objects. Hidden challenges are omitted for non-admin users, and challenges with a future `releaseTime` are omitted until their release timestamp.

```json
{
  "kind": "goodChallenges",
  "message": "The retrieval of challenges was successful.",
  "data": []
}
```

| Field | Type | Notes |
| --- | --- | --- |
| `id` | `string{:ts}` | Challenge ID. |
| `name` | `string{:ts}` | Display name. |
| `description` | `string{:ts}` | Markdown challenge description. |
| `category` | `string{:ts}` | Category name. |
| `author` | `string{:ts}` | Author display name. |
| `files` | `ChallengeFileV2[]{:ts}` | Downloadable files. See [`ChallengeFileV2`](/api/responses/#challengefilev2). |
| `points` | `number{:ts}` | Current score after dynamic scoring. |
| `solves` | `number{:ts}` | Current solve count. |
| `sortWeight` | `number \| null{:ts}` | Optional ordering weight. |
| `instancerLifetime` | `number \| null{:ts}` | Instance timeout in milliseconds when the challenge has instancer config and an instancer provider is enabled. |
| `instancerExtendable` | `boolean{:ts}` | `false{:ts}` only when challenge config explicitly disables extension. |
| `adminBotInputs` | `Record<string, RegexRule> \| null \| undefined{:ts}` | Participant input schema for admin bot challenges. See [`RegexRule`](/api/responses/#regexrule). |
| `hasFlag` | `boolean{:ts}` | Indicates whether the challenge has a flag configured. |

**Errors**: `<red>401 badNotStarted</red>`.

### `<route>GET /api/v2/challs/:id/solves</route>`

Returns paginated solve history for one challenge.

| Property       | Value                                 |
| -------------- | ------------------------------------- |
| Auth           | Optional + bypass (`challsRead{:ts}`) |
| Gate           | Started                               |
| Permissions    | -                                     |
| Captcha action | -                                     |
| Rate limit     | -                                     |

**Path params**

| Field | Type          | Required | Notes         |
| ----- | ------------- | -------- | ------------- |
| `id`  | `string{:ts}` | Yes      | Challenge ID. |

**Query**

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `limit` | `number{:ts}` | Yes | Integer greater than or equal to `1`. The config enforces the maximum. |
| `offset` | `number{:ts}` | Yes | Integer greater than or equal to `0`. The config enforces the maximum. |

**Response**

`<green>200 goodChallengeSolves</green>`:

```json
{
  "kind": "goodChallengeSolves",
  "message": "The challenge solves were retrieved.",
  "data": {
    "solves": [
      {
        "id": "solve-id",
        "createdAt": 1710000000000,
        "userId": "team-id",
        "userName": "otter-sec",
        "userAvatarUrl": null,
        "userCountryCode": "US",
        "userStatusText": null,
        "globalPlace": 3,
        "division": "open",
        "divisionPlace": 1,
        "bloodIndex": 0
      }
    ],
    "mySolvePosition": 12
  }
}
```

| Field | Type | Notes |
| --- | --- | --- |
| `solves[]` | `ChallengeSolveV2[]{:ts}` | Solve rows ordered by solve time. |
| `solves[].bloodIndex` | `number \| null{:ts}` | `0`, `1`, or `2` for the first three solves. Otherwise `null{:ts}`. |
| `mySolvePosition` | `number \| null{:ts}` | The authenticated user's solve position for this challenge when optional auth is present. |

**Errors**: `<red>404 badChallenge</red>`, `<red>401 badNotStarted</red>`.

## V1 compatibility

V1 routes remain available for legacy clients. Behavior matches V2 except where noted below. Shared response and error semantics apply unless overridden.

### `<route>GET /api/v1/challs</route>`

Differs from V2:

- `<green>200 goodChallenges</green>` data omits V2 integration metadata (`instancerLifetime`, `instancerExtendable`, `adminBotInputs`, `hasFlag`) and uses a simpler `files` shape.

The V1 challenge list returns the same core fields without V2 integration metadata:

| Field         | Type                                   |
| ------------- | -------------------------------------- |
| `id`          | `string{:ts}`                          |
| `name`        | `string{:ts}`                          |
| `description` | `string{:ts}`                          |
| `category`    | `string{:ts}`                          |
| `author`      | `string{:ts}`                          |
| `files`       | `{ name: string, url: string }[]{:ts}` |
| `points`      | `number{:ts}`                          |
| `solves`      | `number{:ts}`                          |
| `sortWeight`  | `number \| null \| undefined{:ts}`     |

### `<route>GET /api/v1/challs/:id/solves</route>`

Differs from V2:

- `<green>200 goodChallengeSolves</green>` data returns `{ solves }` without `mySolvePosition`, and each solve row carries only `id`, `createdAt`, `userId`, and `userName` (no avatar, country, status, places, or `bloodIndex`).

```json
{
  "kind": "goodChallengeSolves",
  "message": "The challenge solves were retrieved.",
  "data": {
    "solves": [
      {
        "id": "solve-id",
        "createdAt": 1710000000000,
        "userId": "team-id",
        "userName": "otter-sec"
      }
    ]
  }
}
```

### `<route>POST /api/v1/challs/:id/submit</route>`

V1-only. Submits a flag for the authenticated team. There is no V2 equivalent.

| Property | Value |
| --- | --- |
| Auth | Required |
| Gate | Started + not ended |
| Permissions | - |
| Captcha action | - |
| Rate limit | Burst `5`, refill window `25000` ms, scoped to user and challenge. See [`<route>/api#rate-limits</route>`](/api/#rate-limits). |

**Path params**

| Field | Type          | Required | Notes         |
| ----- | ------------- | -------- | ------------- |
| `id`  | `string{:ts}` | Yes      | Challenge ID. |

**Request body**

```json
{
  "flag": "flag{example}"
}
```

| Field  | Type          | Required | Notes                  |
| ------ | ------------- | -------- | ---------------------- |
| `flag` | `string{:ts}` | Yes      | Maximum length `1024`. |

**Response**

`<green>200 goodFlag</green>` (no data) when the submitted flag is correct. The route records a solve for the team.

The route records a submission audit row for both correct and incorrect attempts. Admins can read those rows through [`<route>/api/v2/admin/submissions</route>`](/api/admin/).

**Errors**: `<red>400 badBody</red>`, `<red>404 badChallenge</red>`, `<red>400 badFlag</red>`, `<red>409 badAlreadySolvedChallenge</red>`, `<red>401 badNotStarted</red>`, `<red>401 badEnded</red>`, `<red>429 badRateLimit</red>`.

## Scoring behavior

Challenge point values are computed by the configured [scoring provider](/providers/scores/). The public challenge list returns the current point value, not the configured `points.min` or `points.max` range. Challenge updates and solve mutations trigger leaderboard recalculation.

## Visibility behavior

Non-admin users only see challenges that are not hidden and whose release time has passed. Admin users with `challsRead{:ts}` can bypass the CTF start gate through the same public routes, but hidden and scheduled release handling is still applied by the challenge service according to the route implementation.
