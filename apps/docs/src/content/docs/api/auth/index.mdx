Authentication in rCTF is team-based. A team is the account that owns the name, email address, score, solves, members, and settings. Browser sessions and API clients act on behalf of a team by sending an auth token in the `Authorization` header.

The API has both V1 and V2 route contracts. They are available together, and a client may need routes from both versions. V2 routes exist where the contract needed to change or where new behavior was added. V1 routes remain current when there is no V2 replacement.

Most authenticated API requests use the `Authorization: Bearer <dim><auth-token></dim>` header.

The other token kinds are used while setting up, recovering, or changing a team account.

| Token kind    | Lifetime       | Used by                                                         |
| ------------- | -------------- | --------------------------------------------------------------- |
| `Auth`        | No expiry      | `Authorization: Bearer <dim><auth-token></dim>` on user routes. |
| `Team`        | No expiry      | Account recovery, login, and token verification.                |
| `Verify`      | `loginTimeout` | Email updates and pending registrations. One-time.              |
| `CtftimeAuth` | `loginTimeout` | CTFtime registration and login handoff.                         |

Tokens are AES-256-GCM encrypted with `tokenKey`. Rotating `tokenKey` invalidates any auth, team, verify, or CTFtime handoff token issued before the rotation.

Verify tokens also depend on a one-time Redis marker. The encrypted token can still decrypt successfully after the marker has been used or expired, but the verification request will not complete.

## Version choice

Use the V2 route when both V1 and V2 exist for the same action. The V2 contract carries the newer captcha field name and richer response data. Use V1 routes for actions that do not have a V2 equivalent, such as logging in with a team token or testing an auth token.
