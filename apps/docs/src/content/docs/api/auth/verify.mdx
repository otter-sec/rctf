Consumes a verification-style token and turns it into the next account state. The same endpoint handles pending registrations, account recovery with a team token, and email-change confirmations.

Use the V2 route for new clients. It returns the wider registration response when a pending registration is completed.

<Tabs items={['V2', 'V1']} sync="verify-version">
  <Tab>

`<route>POST /api/v2/auth/verify</route>` can complete a pending registration, mint an auth token from a team token, or confirm an email change.

| Token kind | Success kind | Effect |
| --- | --- | --- |
| Pending registration | `<response>200 goodRegisterV2</response>` | Creates the team, returns both tokens, and claims the verification row. |
| Team | `<response>200 goodVerify</response>` | Returns a fresh `authToken`. |
| Verify | `<response>200 goodEmailSet</response>` | Updates the team email after re-checking division ACLs. |

For pending registrations, the response body matches the immediate-success shape from [register a team](/api/auth/register/). Team tokens are recovery or login credentials, so they only return a fresh `authToken`. Email-change tokens do not return data after the email is updated.

  </Tab>

  <Tab>

`<route>POST /api/v1/auth/verify</route>` can complete a pending registration, mint an auth token from a team token, or confirm an email change.

| Token kind | Success kind | Effect |
| --- | --- | --- |
| Pending registration | `<response>200 goodRegister</response>` | Creates the team, returns `authToken`, and claims the verification row. |
| Team | `<response>200 goodVerify</response>` | Returns a fresh `authToken`. |
| Verify | `<response>200 goodEmailSet</response>` | Updates the team email after re-checking division ACLs. |

Pending-registration and team-token flows both return a fresh auth token. Email-change tokens do not return data after the email is updated.

  </Tab>
</Tabs>

Verification can fail even when the token decrypts correctly. Verify tokens are one-time and time-limited, and email-change tokens still re-check division ACLs and duplicate-email constraints at the moment they are consumed.
