Admin routes require a user auth token with the listed permission bits, except for admin bot service routes, which use the shared admin bot bearer token. Routes are grouped by resource. Permissions, captcha actions, and rate limits referenced in property tables are defined on the [API overview](/api/).

## Permissions

| Permission | Required by |
| --- | --- |
| `challsRead{:ts}` | Reading admin challenge data, reading upload state, reading instancer schema, reading admin bot status, and reading admin user detail with solve history. |
| `challsWrite{:ts}` | Updating challenges and uploading files. |
| `challsSolveWrite{:ts}` | Deleting solves. |
| `usersWrite{:ts}` | Listing teams, filtering teams, mutating teams, generating team tokens, managing pending verifications, and reading submission audit logs. |
| `settingsWrite{:ts}` | Reading and updating runtime settings. |

Routes that list multiple permission bits require all of them.

## Challenge routes

### `<route>GET /api/v2/admin/challs</route>`

Lists every challenge with full admin fields, including hidden challenges and unreleased challenges.

| Property       | Value             |
| -------------- | ----------------- |
| Auth           | Required          |
| Gate           | None              |
| Permissions    | `challsRead{:ts}` |
| Captcha action | -                 |
| Rate limit     | -                 |

**Response**

`<green>200 goodAdminChallenges</green>`: array of `AdminChallengeV2`. See [`AdminChallengeV2`](#adminchallengev2) below.

**Errors**: `<red>401 badToken</red>`, `<red>403 badPerms</red>`.

### `<route>GET /api/v2/admin/challs/:id</route>`

Returns one admin challenge.

| Property       | Value             |
| -------------- | ----------------- |
| Auth           | Required          |
| Gate           | None              |
| Permissions    | `challsRead{:ts}` |
| Captcha action | -                 |
| Rate limit     | -                 |

**Path params**

| Field | Type          | Required | Notes         |

### `<route>PUT /api/v2/admin/challs/:id</route>`

Creates or updates a challenge. Omitted fields in `data` keep their current value. Setting `instancerConfig{:ts}` or `adminBotConfig{:ts}` to `null{:ts}` clears the integration config.

| Property       | Value              |
| -------------- | ------------------ |
| Auth           | Required           |
| Gate           | None               |
| Permissions    | `challsWrite{:ts}` |
| Captcha action | -                  |
| Rate limit     | -                  |

**Path params**

| Field | Type          | Required | Notes                                 |
| ----- | ------------- | -------- | ------------------------------------- |
| `id`  | `string{:ts}` | Yes      | Challenge ID. Created if not present. |

**Request body**

```json
{
  "data": {
    "name": "web demo",
    "category": "web",
    "author": "organizers",
    "description": "Solve it.",
    "flag": "flag{example}",
    "points": { "min": 100, "max": 500 },
    "tiebreakEligible": true,
    "files": [{ "name": "dist.zip", "url": "/uploads/dist.zip", "size": 12345 }],
    "sortWeight": 10,
    "instancerConfig": null,
    "adminBotConfig": null,
    "hidden": false,
    "releaseTime": null
  }
}
```

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `data` | `Partial<AdminChallengeV2>{:ts}` | Yes | Partial challenge data. Omitted fields keep existing values. |

`instancerConfig{:ts}` is validated against the active instancer provider schema. `adminBotConfig{:ts}` source is validated by the active admin bot provider and stored with a generated revision. Updating a challenge triggers a leaderboard recalculation.

**Response**

`<green>200 goodChallengeUpdate</green>`: updated `AdminChallengeV2`.

**Errors**: `<red>400 badBody</red>`, `<red>401 badToken</red>`, `<red>403 badPerms</red>`, `<red>400 badInstancerConfig</red>`, `<red>400 badAdminBotConfig</red>`.

## Upload routes

### `<route>POST /api/v2/admin/upload</route>`

Uploads one or more files. Files are deduplicated by SHA-256 - uploading the same content twice returns the existing URL.

| Property       | Value              |
| -------------- | ------------------ |
| Auth           | Required           |
| Gate           | None               |
| Permissions    | `challsWrite{:ts}` |
| Captcha action | -                  |
| Rate limit     | -                  |

**Request**

`multipart/form-data` with one or more `files` form fields.

**Response**

`<green>200 goodFilesUpload</green>`: array of upload objects.

| Field  | Type                  | Notes                |
| ------ | --------------------- | -------------------- |
| `name` | `string{:ts}`         | Display filename.    |
| `url`  | `string{:ts}`         | Public download URL. |
| `size` | `number \| null{:ts}` | File size in bytes.  |

**Errors**: `<red>400 badBody</red>`, `<red>401 badToken</red>`, `<red>403 badPerms</red>`, `<red>500 badFilesUpload</red>`.

### `<route>POST /api/v2/admin/upload/query</route>`

Checks whether files already exist by SHA-256 without uploading them.

| Property       | Value             |
| -------------- | ----------------- |
| Auth           | Required          |
| Gate           | None              |
| Permissions    | `challsRead{:ts}` |
| Captcha action | -                 |
| Rate limit     | -                 |

**Request body**

```json
{
  "uploads": [{ "sha256": "<hex-digest>", "name": "dist.zip" }]
}
```

| Field              | Type          | Required | Notes                       |
| ------------------ | ------------- | -------- | --------------------------- |
| `uploads[].sha256` | `string{:ts}` | Yes      | Hex-encoded SHA-256 digest. |
| `uploads[].name`   | `string{:ts}` | Yes      | Display filename.           |

**Response**

`<green>200 goodUploadsQuery</green>`: array with one entry per query - each entry has nullable `url` and nullable `size`. A `null{:ts}` URL means the file is not yet uploaded.

**Errors**: `<red>400 badBody</red>`, `<red>401 badToken</red>`, `<red>403 badPerms</red>`.

## Team routes

### `<route>GET /api/v2/admin/users</route>`

Lists teams. Query parameters control pagination, sorting, and search.

| Property       | Value             |
| -------------- | ----------------- |
| Auth           | Required          |
| Gate           | None              |
| Permissions    | `usersWrite{:ts}` |
| Captcha action | -                 |
| Rate limit     | -                 |

**Query**

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `limit` | `number{:ts}` | Yes | Integer from `1` through `100`. |
| `offset` | `number{:ts}` | Yes | Integer ≥ `0`. |
| `search` | `string{:ts}` | No | 1–100 characters. |
| `sortBy` | `AdminTeamSortBy{:ts}` | No | `createdAt`, `team`, `email`, `division`, `score`, `solves`, or `status`. |
| `sortOrder` | `SortOrder{:ts}` | No | `asc` or `desc`. |

**Response**

`<green>200 goodAdminUsers</green>`: `{ total: number, users: AdminUserListItem[] }`. Each list item includes ID, name, email, division, perms, banned state, score, solve count, avatar URL, country code, status text, and `createdAt`.

### `<route>POST /api/v2/admin/users</route>`

Lists teams with rich filters. Same query parameters as `<route>GET</route>`, with filter object in the body.

| Property       | Value             |
| -------------- | ----------------- |

**Errors**: `<red>400 badBody</red>`, `<red>401 badToken</red>`, `<red>403 badPerms</red>`.

## V1 compatibility

### Challenges

#### `<route>GET /api/v1/admin/challs</route>`

Same as [`<route>GET /api/v2/admin/challs</route>`](#get-apiv2adminchalls), but V1 admin challenge data omits `hidden`, `releaseTime`, `instancerConfig`, `adminBotConfig`, and file `size`.

#### `<route>PUT /api/v1/admin/challs/:id</route>`

Same as [`<route>PUT /api/v2/admin/challs/:id</route>`](#put-apiv2adminchallsid), with the V1 challenge data shape (no `hidden`, `releaseTime`, `instancerConfig`, `adminBotConfig`, or file `size`).

#### `<route>DELETE /api/v1/admin/challs/:id</route>`

V1-only. Deletes a challenge and all its solves. Triggers a leaderboard recalculation. No V2 equivalent - V2 deletes via challenge mutation in the admin panel.

| Property       | Value              |
| -------------- | ------------------ |
| Auth           | Required           |
| Gate           | None               |
| Permissions    | `challsWrite{:ts}` |
| Captcha action | -                  |
| Rate limit     | -                  |

**Path params**

| Field | Type          | Required | Notes         |
| ----- | ------------- | -------- | ------------- |
| `id`  | `string{:ts}` | Yes      | Challenge ID. |

**Response**

`<green>200 goodChallengeDelete</green>` (no data).

**Errors**: `<red>401 badToken</red>`, `<red>403 badPerms</red>`, `<red>404 badChallenge</red>`.

### Uploads

#### `<route>POST /api/v1/admin/upload</route>`

Differs from V2: accepts JSON with base64-encoded data URIs instead of `multipart/form-data`.

**Request body**

```json
{
  "files": [{ "name": "dist.zip", "data": "data:application/zip;base64,..." }]
}
```

V1 upload responses omit the `size` field.

**Errors**: `<red>400 badBody</red>`, `<red>401 badToken</red>`, `<red>403 badPerms</red>`, `<red>400 badDataUri</red>`, `<red>500 badFilesUpload</red>`.

#### `<route>POST /api/v1/admin/upload/query</route>`

Same as [`<route>POST /api/v2/admin/upload/query</route>`](#post-apiv2adminuploadquery), but V1 upload query responses include nullable `url` only - no `size`.
