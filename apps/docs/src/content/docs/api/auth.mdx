Authentication is team-based. A team owns one account, one team token, and any number of auth tokens. Every route on this page is public except `<route>GET /api/v1/auth/test</route>`, which validates an auth token.

## V2 routes

### `<route>POST /api/v2/auth/register</route>`

Registers a team. Returns auth tokens immediately when verification is not required, otherwise sends a verification email.

| Property       | Value                             |
| -------------- | --------------------------------- |
| Auth           | Public                            |
| Gate           | None                              |
| Permissions    | -                                 |
| Captcha action | `register{:ts}` (when configured) |
| Rate limit     | -                                 |

**Request body**

```json
{
  "name": "otter-sec",
  "email": "team@example.com",
  "ctftimeToken": "optional-ctftime-token",
  "captchaCode": "optional-captcha-code"
}
```

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `name` | `string{:ts}` | Yes | 2–64 printable ASCII characters. |
| `email` | `string{:ts}` | Conditional | Required when `ctftimeToken` is omitted. |
| `ctftimeToken` | `string{:ts}` | Conditional | Required when `email` is omitted. Returns `<red>404 badEndpoint</red>` when CTFtime is not configured. |
| `captchaCode` | `string{:ts}` | No | Checked only when captcha protects `register{:ts}`. |

The route assigns the team to the default allowed division based on email ACLs. CTFtime registrations bypass ACLs entirely.

**Response**

When registration completes immediately (CTFtime auth, or no email provider is configured), returns `<green>200 goodRegisterV2</green>`:

```json
{
  "kind": "goodRegisterV2",
  "message": "The user was created.",
  "data": {
    "authToken": "<auth-token>",
    "teamToken": "<team-token>"
  }
}
```

When email verification is required, returns `<green>200 goodVerifySent</green>` (no data) and sends a verification email.

**Errors**: `<red>400 badBody</red>`, `<red>400 badEmail</red>`, `<red>400 badName</red>`, `<red>409 badKnownEmail</red>`, `<red>409 badKnownName</red>`, `<red>409 badKnownCtftimeId</red>`, `<red>403 badCompetitionNotAllowed</red>`, `<red>403 badDivisionNotAllowed</red>`, `<red>400 badRegistrationsDisabled</red>`, `<red>403 badCaptcha</red>`, `<red>404 badEndpoint</red>`, `<red>401 badCtftimeToken</red>`.

### `<route>POST /api/v2/auth/verify</route>`

Consumes a verification token to complete a pending registration, log in with a team token, or finalize an email change.

| Property       | Value  |
| -------------- | ------ |
| Auth           | Public |
| Gate           | None   |
| Permissions    | -      |
| Captcha action | -      |
| Rate limit     | -      |

**Request body**

```json
{
  "verifyToken": "<token>"
}
```

| Field         | Type          | Required | Notes                                        |
| ------------- | ------------- | -------- | -------------------------------------------- |
| `verifyToken` | `string{:ts}` | Yes      | Pending-registration, team, or verify token. |

**Response**

Behavior branches on the token kind:

| Token kind | Success kind | Effect |
| --- | --- | --- |
| Pending registration | `<green>200 goodRegisterV2</green>` | Creates the team and returns `authToken` + `teamToken`. Claims the verification row. |
| Team | `<green>200 goodVerify</green>` | Returns a fresh `authToken`. |
| Verify | `<green>200 goodEmailSet</green>` | Updates the team email after re-checking division ACLs. |

`<green>200 goodRegisterV2</green>` data matches the [register response](#post-apiv2authregister). `<green>200 goodVerify</green>` returns `{ authToken: string }`. `<green>200 goodEmailSet</green>` returns no data.

**Errors**: `<red>400 badBody</red>`, `<red>401 badTokenVerification</red>`, `<red>403 badEmailChangeDivision</red>`, `<red>409 badKnownEmail</red>`.

### `<route>POST /api/v2/auth/recover</route>`

Sends an account-recovery email containing a team token. Always returns `<green>200 goodVerifySent</green>`, even when the email is unknown, to prevent account enumeration.

| Property       | Value                            |
| -------------- | -------------------------------- |
| Auth           | Public                           |
| Gate           | None                             |
| Permissions    | -                                |
| Captcha action | `recover{:ts}` (when configured) |
| Rate limit     | -                                |

**Request body**

```json
{
  "email": "team@example.com",
  "captchaCode": "optional-captcha-code"
}
```

| Field         | Type          | Required | Notes                                              |
| ------------- | ------------- | -------- | -------------------------------------------------- |
| `email`       | `string{:ts}` | Yes      | Recovery destination.                              |
| `captchaCode` | `string{:ts}` | No       | Checked only when captcha protects `recover{:ts}`. |

**Response**

`<green>200 goodVerifySent</green>` (no data).

**Errors**: `<red>400 badBody</red>`, `<red>400 badEmail</red>`, `<red>404 badEndpoint</red>` (no email provider configured), `<red>403 badCaptcha</red>`.

### `<route>GET /api/v2/auth/verify-info</route>`

Previews a verification token without consuming it. Useful for showing context on the verification page before the user submits.

| Property       | Value  |
| -------------- | ------ |
| Auth           | Public |
| Gate           | None   |
| Permissions    | -      |
| Captcha action | -      |
| Rate limit     | -      |

**Query**

| Field   | Type          | Required | Notes                                        |
| ------- | ------------- | -------- | -------------------------------------------- |
| `token` | `string{:ts}` | Yes      | Pending-registration, team, or verify token. |

**Response**

`<green>200 goodVerifyInfo</green>`:

```json
{
  "kind": "goodVerifyInfo",
  "message": "The verification info was retrieved.",
  "data": {
    "kind": "register",
    "email": "team@example.com",
    "name": "otter-sec"
  }
}
```

| Field | Type | Notes |
| --- | --- | --- |
| `data.kind` | `"register" \| "team" \| "update"{:ts}` | Token purpose. |
| `data.email` | `string \| null{:ts}` | `null{:ts}` for team-token recovery previews. |
| `data.name` | `string \| undefined{:ts}` | Present for pending-registration tokens. |

**Errors**: `<red>400 badBody</red>`, `<red>401 badTokenVerification</red>`.

## V1 compatibility

V1 routes remain available for legacy clients. Behavior matches V2 except where noted below. Shared response and error semantics apply unless overridden.

### `<route>POST /api/v1/auth/register</route>`

Differs from V2:

- Captcha field is `recaptchaCode` instead of `captchaCode`.
- Immediate success returns `<green>200 goodRegister</green>` with `{ authToken: string }` only - no `teamToken`.

### `<route>POST /api/v1/auth/login</route>`

V1-only. Exchanges a team token or a CTFtime auth token for an auth token. There is no V2 equivalent for CTFtime login; V2 clients use [`<route>POST /api/v2/auth/verify</route>`](#post-apiv2authverify) with a team token, or the CTFtime callback flow.

| Property       | Value  |
| -------------- | ------ |
| Auth           | Public |
| Gate           | None   |
| Permissions    | -      |
| Captcha action | -      |
| Rate limit     | -      |

**Request body**

```json
{
  "teamToken": "<team-token>",
  "ctftimeToken": "<ctftime-token>"
}
```

At least one of `teamToken` or `ctftimeToken` must be present. A team token is parsed as `TokenKind.Team{:ts}`. A CTFtime token is parsed as `TokenKind.CtftimeAuth{:ts}` and matched against a team linked to that CTFtime ID.

**Response**

`<green>200 goodLogin</green>`: `{ authToken: string }`.

**Errors**: `<red>400 badBody</red>`, `<red>401 badTokenVerification</red>`, `<red>401 badCtftimeToken</red>`, `<red>404 badUnknownUser</red>`.

### `<route>POST /api/v1/auth/verify</route>`

Differs from V2: the pending-registration path returns `<green>200 goodRegister</green>` with `{ authToken: string }` only.

### `<route>POST /api/v1/auth/recover</route>`

Differs from V2: captcha field is `recaptchaCode` instead of `captchaCode`.

### `<route>GET /api/v1/auth/test</route>`

Validates the supplied auth token. Returns `<green>200 goodToken</green>` (no data) when the token is valid, `<red>401 badToken</red>` otherwise. Useful for client-side session checks.

| Property       | Value    |
| -------------- | -------- |
| Auth           | Required |
| Gate           | None     |
| Permissions    | -        |
| Captcha action | -        |
| Rate limit     | -        |

## Token lifecycle

All tokens are AES-256-GCM encrypted with `tokenKey`. Rotating `tokenKey` invalidates every existing token.

| Token kind    | Lifetime       | Used by                                              |
| ------------- | -------------- | ---------------------------------------------------- |
| `Auth`        | No expiry      | `Authorization: Bearer <auth-token>` on user routes. |
| `Team`        | No expiry      | Recovery, V1 login, and V2 verify with a team token. |
| `Verify`      | `loginTimeout` | Email updates and pending registrations. One-time.   |
| `CtftimeAuth` | `loginTimeout` | CTFtime registration and V1 login handoff.           |

Verify tokens additionally require a one-time Redis marker, so a valid encrypted token can still return `<red>401 badTokenVerification</red>` after the marker has been consumed or expired.
