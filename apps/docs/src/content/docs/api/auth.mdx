Authentication is team-based. A team owns one account, one team token, and any number of auth tokens. Every route on this page is public except `GET /api/v1/auth/test`, which validates an auth token.

## V2 routes

### `POST /api/v2/auth/register`

Registers a team. Returns auth tokens immediately when verification is not required, otherwise sends a verification email.

| Property | Value |
| --- | --- |
| Auth | Public |
| Gate | None |
| Permissions | — |
| Captcha action | `register{:ts}` (when configured) |
| Rate limit | — |

**Request body**

```json
{
  "name": "maple bacon",
  "email": "team@example.com",
  "ctftimeToken": "optional-ctftime-token",
  "captchaCode": "optional-captcha-code"
}
```

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `name` | `string{:ts}` | Yes | 2–64 printable ASCII characters. |
| `email` | `string{:ts}` | Conditional | Required when `ctftimeToken` is omitted. |
| `ctftimeToken` | `string{:ts}` | Conditional | Required when `email` is omitted. Returns `badEndpoint` when CTFtime is not configured. |
| `captchaCode` | `string{:ts}` | No | Checked only when captcha protects `register{:ts}`. |

The route assigns the team to the default allowed division based on email ACLs. CTFtime registrations bypass ACLs entirely.

**Response**

When registration completes immediately (CTFtime auth, or no email provider is configured), returns `goodRegisterV2` (200):

```json
{
  "kind": "goodRegisterV2",
  "message": "The user was created.",
  "data": {
    "authToken": "<auth-token>",
    "teamToken": "<team-token>"
  }
}
```

When email verification is required, returns `goodVerifySent` (no data) and sends a verification email.

**Errors**: `badBody`, `badEmail`, `badName`, `badKnownEmail`, `badKnownName`, `badKnownCtftimeId`, `badCompetitionNotAllowed`, `badDivisionNotAllowed`, `badRegistrationsDisabled`, `badCaptcha`, `badEndpoint`, `badCtftimeToken`.

### `POST /api/v2/auth/verify`

Consumes a verification token to complete a pending registration, log in with a team token, or finalize an email change.

| Property | Value |
| --- | --- |
| Auth | Public |
| Gate | None |
| Permissions | — |
| Captcha action | — |
| Rate limit | — |

**Request body**

```json
{
  "verifyToken": "<token>"
}
```

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `verifyToken` | `string{:ts}` | Yes | Pending-registration, team, or verify token. |

**Response**

Behavior branches on the token kind:

| Token kind | Success kind | Effect |
| --- | --- | --- |
| Pending registration | `goodRegisterV2` | Creates the team and returns `authToken` + `teamToken`. Claims the verification row. |
| Team | `goodVerify` | Returns a fresh `authToken`. |
| Verify | `goodEmailSet` | Updates the team email after re-checking division ACLs. |

`goodRegisterV2` data matches the [register response](#post-apiv2authregister). `goodVerify` returns `{ authToken: string }`. `goodEmailSet` returns no data.

**Errors**: `badBody`, `badTokenVerification`, `badEmailChangeDivision`, `badKnownEmail`.

### `POST /api/v2/auth/recover`

Sends an account-recovery email containing a team token. Always returns `goodVerifySent`, even when the email is unknown, to prevent account enumeration.

| Property | Value |
| --- | --- |
| Auth | Public |
| Gate | None |
| Permissions | — |
| Captcha action | `recover{:ts}` (when configured) |
| Rate limit | — |

**Request body**

```json
{
  "email": "team@example.com",
  "captchaCode": "optional-captcha-code"
}
```

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `email` | `string{:ts}` | Yes | Recovery destination. |
| `captchaCode` | `string{:ts}` | No | Checked only when captcha protects `recover{:ts}`. |

**Response**

`goodVerifySent` (no data).

**Errors**: `badBody`, `badEmail`, `badEndpoint` (no email provider configured), `badCaptcha`.

### `GET /api/v2/auth/verify-info`

Previews a verification token without consuming it. Useful for showing context on the verification page before the user submits.

| Property | Value |
| --- | --- |
| Auth | Public |
| Gate | None |
| Permissions | — |
| Captcha action | — |
| Rate limit | — |

**Query**

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `token` | `string{:ts}` | Yes | Pending-registration, team, or verify token. |

**Response**

`goodVerifyInfo` (200):

```json
{
  "kind": "goodVerifyInfo",
  "message": "The verification info was retrieved.",
  "data": {
    "kind": "register",
    "email": "team@example.com",
    "name": "maple bacon"
  }
}
```

| Field | Type | Notes |
| --- | --- | --- |
| `data.kind` | `"register" \| "team" \| "update"{:ts}` | Token purpose. |
| `data.email` | `string \| null{:ts}` | `null{:ts}` for team-token recovery previews. |
| `data.name` | `string \| undefined{:ts}` | Present for pending-registration tokens. |

**Errors**: `badBody`, `badTokenVerification`.

## V1 compatibility

V1 routes remain available for legacy clients. Behavior matches V2 except where noted below. Shared response and error semantics apply unless overridden.

### `POST /api/v1/auth/register`

Differs from V2:

- Captcha field is `recaptchaCode` instead of `captchaCode`.
- Immediate success returns `goodRegister` with `{ authToken: string }` only — no `teamToken`.

### `POST /api/v1/auth/login`

V1-only. Exchanges a team token or a CTFtime auth token for an auth token. There is no V2 equivalent for CTFtime login; V2 clients use [`POST /api/v2/auth/verify`](#post-apiv2authverify) with a team token, or the CTFtime callback flow.

| Property | Value |
| --- | --- |
| Auth | Public |
| Gate | None |
| Permissions | — |
| Captcha action | — |
| Rate limit | — |

**Request body**

```json
{
  "teamToken": "<team-token>",
  "ctftimeToken": "<ctftime-token>"
}
```

At least one of `teamToken` or `ctftimeToken` must be present. A team token is parsed as `TokenKind.Team{:ts}`. A CTFtime token is parsed as `TokenKind.CtftimeAuth{:ts}` and matched against a team linked to that CTFtime ID.

**Response**

`goodLogin` (200): `{ authToken: string }`.

**Errors**: `badBody`, `badTokenVerification`, `badCtftimeToken`, `badCtftimeNoExists`, `badEndpoint` (CTFtime not configured).

### `POST /api/v1/auth/verify`

Differs from V2: the pending-registration path returns `goodRegister` with `{ authToken: string }` only.

### `POST /api/v1/auth/recover`

Differs from V2: captcha field is `recaptchaCode` instead of `captchaCode`.

### `GET /api/v1/auth/test`

Validates the supplied auth token. Returns `goodToken` (no data) when the token is valid, `badToken` otherwise. Useful for client-side session checks.

| Property | Value |
| --- | --- |
| Auth | Required |
| Gate | None |
| Permissions | — |
| Captcha action | — |
| Rate limit | — |

## Token lifecycle

All tokens are AES-256-GCM encrypted with `tokenKey`. Rotating `tokenKey` invalidates every existing token.

| Token kind | Lifetime | Used by |
| --- | --- | --- |
| `Auth` | No expiry | `Authorization: Bearer <auth-token>` on user routes. |
| `Team` | No expiry | Recovery, V1 login, and V2 verify with a team token. |
| `Verify` | `loginTimeout` | Email updates and pending registrations. One-time. |
| `CtftimeAuth` | `loginTimeout` | CTFtime registration and V1 login handoff. |

Verify tokens additionally require a one-time Redis marker, so a valid encrypted token can still return `badTokenVerification` after the marker has been consumed or expired.
