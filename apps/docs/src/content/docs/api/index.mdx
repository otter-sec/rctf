The rCTF API is a JSON REST API mounted under `[2m/api[22m{:ansi}`. Route definitions live in `@rctf/types`, and the API server registers each route by prefixing the typed path with `[2m/api[22m{:ansi}`.

## Versions

Both API versions are available at the same time:

| Version | Prefix | Status | Use |
| --- | --- | --- | --- |
| V1 | `[2m/api/[22mv1{:ansi}` | Supported | Legacy clients and compatibility with the original rCTF API. |
| V2 | `[2m/api/[22mv2{:ansi}` | Recommended | New integrations, richer response data, runtime settings, avatars, instancers, admin bot jobs, and admin search APIs. |

V2 keeps many response `kind` strings identical to v1 for compatible concepts. The response data may still be wider in v2.

## Typed route contract

The `@rctf/types` package exports every public route definition, response definition, enum, and helper type:

```ts showLineNumbers=false
import {
  GetChallengesRouteV2,
  type RouteBodyInput,
  type RouteQueryInput,
  type RouteSuccessResponse,
} from '@rctf/types'

type Query = RouteQueryInput<typeof GetChallengesRouteV2>
type Response = RouteSuccessResponse<typeof GetChallengesRouteV2>
```

`RouteBodyInput<T>{:ts}` describes the client-side input shape before zod coercions and transforms. `RouteQueryInput<T>{:ts}` and `RouteParamsInput<T>{:ts}` do the same for query strings and path params.

## Authentication

User-authenticated routes require an auth token in the `Authorization` header:

```text
Authorization: Bearer <auth-token>
```

Service-authenticated admin bot routes also use the `Authorization` header, but the token is the shared admin bot service secret from `adminBot.provider.options.secretKey`.

| Auth mode | Behavior |
| --- | --- |
| Public | No token is required. Some public routes still accept optional auth so admins can bypass start-time gates. |
| Optional | The route works without a token, but an auth token may affect the response. |
| Required | A missing or invalid token returns `[2;31m401 [22mbadToken[0m{:ansi}`. |
| Permissioned | The user token must be valid and the user must have every required permission bit. Missing bits return `[2;31m403 [22mbadPerms[0m{:ansi}`. |
| Service | The shared service bearer token must match the configured provider secret. |

### Token types

| Token | Lifetime | Used by |
| --- | --- | --- |
| Auth | No expiry | `Authorization: Bearer <auth-token>` on user routes. |
| Team | No expiry | Account recovery and v1 login. V2 registration returns it directly when registration completes without email verification. |
| Verify | `loginTimeout` | Email update verification. |
| CTFtime auth | `loginTimeout` | CTFtime registration and login handoff. |

Tokens are encrypted with AES-GCM using `tokenKey`. Changing `tokenKey` invalidates existing tokens.

## Response envelope

Most API routes return a JSON envelope:

```json
{
  "kind": "goodChallenges",
  "message": "The retrieval of challenges was successful.",
  "data": []
}
```

Routes whose response definition has no data schema omit the `data` field:

```json
{
  "kind": "goodFlag",
  "message": "The flag is correct."
}
```

The v1 CTFtime leaderboard route is the only typed route that intentionally returns the data body directly. `[36mGET[0m [2m/api/[22mv1[2m/[22mintegrations[2m/[22mctftime[2m/[22mleaderboard{:ansi}` returns `{ "standings": [...] }` instead of an rCTF envelope.

## Request validation

JSON routes parse the request body as JSON when a body schema exists. Form routes parse `multipart/form-data`. Validation errors return `[2;31m400 [22mbadBody[0m{:ansi}` with a machine-readable `reason` string:

```json
{
  "kind": "badBody",
  "message": "The request body does not meet requirements.",
  "data": {
    "reason": "query:limit: Too small: expected number to be >=1"
  }
}
```

The reason prefix identifies the source as `body`, `query`, or `params`. Malformed JSON returns `[2;31m400 [22mbadJson[0m{:ansi}`. Malformed form data returns `[2;31m400 [22mbadBody[0m{:ansi}` with `body:formData:malformed`.

## Permissions

Admin routes use bitmask permissions from `Permissions{:ts}`:

| Permission | Bit | Routes |
| --- | --- | --- |
| `challsRead{:ts}` | `1` | Admin challenge reads, instancer schema, admin bot status, challenge start-time bypass. |
| `challsWrite{:ts}` | `2` | Challenge updates, challenge deletion in v1, file upload. |
| `leaderboardRead{:ts}` | `4` | CTFtime leaderboard and leaderboard start-time bypass. |
| `challsSolveWrite{:ts}` | `8` | Solve deletion. |
| `usersWrite{:ts}` | `16` | Admin user lists, user mutation, pending verification management, submission audit logs. |
| `settingsWrite{:ts}` | `32` | Runtime settings reads and updates. |

When a route lists multiple permission bits, the user must have all of them.

## Timing gates

Routes marked as start-gated return `[2;31m401 [22mbadNotStarted[0m{:ansi}` before `startTime`. Admin users can bypass that gate when the route defines a bypass permission and their token has the required bit.

`[32mPOST[0m [2m/api/[22mv1[2m/[22mchalls[2m/[22m:id[2m/[22msubmit{:ansi}` is also end-gated. It returns `[2;31m401 [22mbadEnded[0m{:ansi}` after `endTime`.

## Captcha

Captcha-protected routes validate a challenge response only when the configured captcha provider marks that action as protected. V1 request bodies use `recaptchaCode`. V2 request bodies use `captchaCode`.

| Action | Routes |
| --- | --- |
| `register{:ts}` | `[32mPOST[0m [2m/api/[22mv1[2m/[22mauth[2m/[22mregister{:ansi}`, `[32mPOST[0m [2m/api/[22mv2[2m/[22mauth[2m/[22mregister{:ansi}`. |
| `recover{:ts}` | `[32mPOST[0m [2m/api/[22mv1[2m/[22mauth[2m/[22mrecover{:ansi}`, `[32mPOST[0m [2m/api/[22mv2[2m/[22mauth[2m/[22mrecover{:ansi}`. |
| `setEmail{:ts}` | `[35mPUT[0m [2m/api/[22mv1[2m/[22musers[2m/[22mme[2m/[22mauth[2m/[22memail{:ansi}`, `[35mPUT[0m [2m/api/[22mv2[2m/[22musers[2m/[22mme[2m/[22mauth[2m/[22memail{:ansi}`. |
| `avatarUpload{:ts}` | `[33mPATCH[0m [2m/api/[22mv2[2m/[22musers[2m/[22mme[2m/[22mavatar{:ansi}`. |
| `instancerStart{:ts}` | `[35mPUT[0m [2m/api/[22mv2[2m/[22mintegrations[2m/[22mchalls[2m/[22m:id[2m/[22minstance{:ansi}`. |

## Rate limits

Rate-limited routes return `[2;31m429 [22mbadRateLimit[0m{:ansi}` with `data.timeLeft`, measured in milliseconds:

| Action | Scope | Bucket |
| --- | --- | --- |
| Flag submission | User and challenge | Burst `5`, refill window `25000` ms. |
| Profile name update | User | Burst `3`, refill window `180000` ms. |
| Avatar upload | User | Burst `2`, refill window `120000` ms. |
| Admin bot submission | User and challenge | Burst `1`, refill window `10000` ms. |
| Leaderboard search | IP address | Burst `3`, refill window `3000` ms. |
