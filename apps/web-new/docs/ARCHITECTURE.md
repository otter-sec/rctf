# web-new architecture

Decisions for the rewrite, with rationale. When implementation contradicts this
document, either fix the code or update the document in the same change.

## Mission

Rewrite `apps/web` with the same fundamental functionality but a codebase built
for readability, concision, and foresight. Native CSS with scoped `<style>`
tags and custom HTML elements instead of Tailwind/shadcn; Zag.js machines
instead of bits-ui primitives. Styling beyond basic layout is explicitly out of
scope until functionality is complete.

## Non-negotiable constraints

These come from the deploy model, the archiver, and public documentation — they
are contracts, not preferences (research: 05, 07, 12):

1. **Same-origin `/api`.** nginx proxies `/api` and `/uploads`; there is no
   client-side backend-URL configuration. All data access goes through
   `fetch('/api/...')` — the archive exporter's `window.fetch` interceptor
   depends on this.
2. **Pure client-side SPA** on `@sveltejs/adapter-static` with
   `fallback: 'index.html'`, `precompress: true`; root layout sets
   `ssr = false`, `csr = true`.
3. **CSP generated by SvelteKit** as a meta tag in prod (script hashes; captcha
   - analytics + storage + YouTube allow-list). `frame-ancestors` stays in
     nginx. Port the existing `svelte.config.ts` CSP block.
4. **Storage keys are contracts** (existing deployments hold them):
   `localStorage['token']`, `localStorage['theme']`,
   `localStorage['rctf:scores:preferences']`, `sessionStorage['ctftimeToken']`,
   `sessionStorage['ctftimeName']`.
5. **Documented route paths must exist**: `/`, `/login`, `/register`,
   `/recover`, `/verify`, `/challenges`, `/scores`, `/profile`, `/profile/:id`,
   `/admin/*`, `/external-auth/authorize`, `/integrations/ctftime/callback`.
6. **Theming token names are public API** (`theming/colors.md`): the Layer C
   semantic tokens (`--background-l0..l5`, `--foreground-l0..l5`, accent /
   destructive / success, gold/silver/bronze/self/nth, `--foreground-first..tenth`,
   prose tokens, per-hue category ramps, `--category-*` indirection) and the
   `data-theme` attribute on `<html>`. Keep the names; the mechanism may change.
7. **Scope bounds** (`meta/things-we-will-not-implement.md`): no blood bonus
   points, no leaderboard freeze, no limited submission attempts, and the
   challenge list + scoreboard stay publicly viewable without login.

## CSS architecture

Modeled on astro-erudite (research: 10 §4 has verbatim artifacts to copy).

```
src/styles/
  reset.css        — erudite's Preflight-derived reset, copied verbatim
  fonts.css        — self-hosted @font-face (keep rCTF's Outfit + Geist Mono)
                     + metric-adjusted local fallbacks (size-adjust/ascent/descent)
  color.css        — the rCTF token contract re-expressed with light-dark();
                     raw palette vendored from Radix Colors (see below)
  typography.css   — fluid type steps (Utopia), constant leading, link rules
  layout.css       — fluid space scale, page grid tokens
  shape.css        — radius/blur tokens
  prose.css        — prose-content element (markdown rendering)
  layers.css       — z-index scale (see Zag rules)
```

Imported once, in order, in the root `+layout.svelte`.

- **Palette: Radix Colors, vendored — not a dependency** (replaces the old
  Tailwind-derived scales). Hex values copied verbatim from the MIT
  `radix-ui/colors` repo into `color.css` as `light-dark()` pairs; only the
  scales/steps we use. Radix's 12-step semantics map directly onto our
  documented ladder: steps 1–5 → `--background-l0..l4`, 6/8 → border/ring,
  11/12 → `--foreground-l1`/`-l0`, steps 3–5 of accent scales → the soft
  accent/destructive/success washes, alpha steps → the category ramps, and
  the literal `gold`/`bronze` scales → scoreboard medals. Layer C token
  _names_ are unchanged (they're public API). Full step-by-step mapping and
  vendoring procedure: research/13-radix-colors.md.
- **Dark mode**: `:root { color-scheme: light dark }` + every color token
  defined once via `light-dark()`. Explicit override:
  `:root[data-theme="light"] { color-scheme: light }` (and dark). Pre-paint
  inline script in `app.html` reads `localStorage.theme` into
  `document.documentElement.dataset.theme`. This preserves the documented
  `data-theme` attribute and `localStorage.theme` while halving token
  definitions (no duplicated dark block).
- **Custom elements as styling hooks**: non-registered elements
  (`nav-countdown`, `challenge-list`, `score-row`, …) replace div-soup and
  give scoped styles collision-free targets. Every custom element gets an
  explicit `display`. Naming: `context-part` kebab-case.
- **Data attributes over classes** for variants and state:
  `data-variant="destructive"`, `toggleAttribute('data-active', bool)`,
  numeric state via CSS custom properties. Zag already emits
  `data-state`/`data-part` — style those directly.
- **Component customization** via CSS custom properties with fallbacks
  (`var(--underline, transparent)` pattern), never style props.
- **Category theming**: the generic `--category-*` tokens are remapped onto a
  hue ramp declaratively via `[data-category-color='<hue>']` blocks in
  `color.css`. A container carries `data-category-color={config.color}`;
  descendants read the generic tokens (custom properties inherit). No JS emits
  category CSS — the old `getCategoryStyle()` string-builder was removed. CSS
  cannot construct a custom-property name from a value, so each of the ten hues
  is an explicit block; keep `CategoryColor` (categories.ts) and those blocks
  in sync.
- **No shadows, max font-weight ~medium, minimal transitions** (documented
  design language). Global reduced-motion kill switch lives in the reset.
- Fluid scales (Utopia `clamp()`) instead of breakpoint ladders; media queries
  use range syntax (`@media (width < 48rem)`) and live at the end of each rule.

## Component tiers

Full per-family mapping with usage counts: research 06. Summary:

**Tier 1 — native HTML/CSS (thin or no wrapper):**
button (one component: `data-variant`/`data-size`, href polymorphism), input,
textarea, checkbox (styled native), label, progress (native element), hr,
card/section (custom elements + scoped CSS, no JS), spinner, empty-state,
badge/chip, search-input, captcha-notice, archived-notice.

**Tier 2 — Zag machines (wrapped once in `lib/ui`):**

- `dialog` — ONE primitive, three CSS presentations: center modal, side sheet
  (mobile nav), bottom drawer (mobile master-detail). Replaces bits dialog +
  sheet + vaul. Drag-to-dismiss is dropped unless it proves missed.
- `menu` — with submenus + checkbox items (admin filters are the acceptance
  test).
- `select`, `combobox` (flag-picker: kills the command + popover families),
  `tooltip` (delay grouping must match old Provider semantics; scores page
  uses skip-delay regions), `tabs`, `accordion` (controlled multi-value for
  collapse/expand-all), `splitter` (two master-detail screens), `avatar`,
  `toast` (group machine + store, hosted in root layout).

**Tier 3 — custom Svelte (ported or new):**
scroll-area (native `overflow:auto` + ported edge-fade logic), virtual-list
(port; keeps tanstack-virtual), schema-form (port ~16 files; logic is
dependency-free, swap leaf inputs), tag-input (port, 106 lines), markdown
(ported pipeline + island hydration of alerts/timers), chart kit (new:
scale/axis/line/bars/hover/tooltip), field/form layout primitives, navigation,
theme-toggle (FOUC-safe), ctf-not-started countdown.

**Deleted dead weight** (zero usage in apps/web): button-group, input-group,
pagination, sidebar (26 files), skeleton, table family, command-dialog.

## Zag.js rules (learned the hard way — research 11)

1. **Reactive props MUST be thunks**: `useMachine(m.machine, () => ({ id, open }))`.
   A plain object silently freezes controlled props. Enforce in review.
2. `const api = $derived(m.connect(service, normalizeProps))` — always
   `$derived`, always spread prop-getters fresh.
3. IDs come from `$props.id()` (Svelte ≥5.20) — never `Math.random()`.
4. Pin all `@zag-js/*` to the same exact 1.x version (v2 is on the horizon;
   do not mix core/machine versions).
5. Vendor `Portal.svelte` (+`Presence.svelte` only if exit animations are
   wanted); keep overlays behind `{#if api.open}`.
6. z-index is ours: `layers.css` defines the scale
   (backdrop < dialog < popover < toast).
7. Use exported `mergeProps` to combine our handlers with prop-getters.
8. Select/combobox `collection` must be rebuilt reactively from fetched items
   (thunk again).
9. Treat `chakra-ui/zag` repo `examples/svelte-ts/` as the source of truth over
   the docs site.

## Data layer

- `lib/api/client.ts`: port of the route-definition `apiRequest` (typed args,
  envelope validation, `Bearer` header, `badToken` auto-clear, captcha
  injection, form-data support, rate-limit toast helper).
- `lib/query/`: per-domain `queryOptions` factories + central `queryKeys`
  hierarchy, same polling policy as today (challenges 30s, leaderboard 30s +
  focus refetch, userSelf 30s/staleTime 5m, clientConfig Infinity). Mutations
  stay thin (`createApiMutation`); invalidation stays caller-side — but
  centralize the recurring trios as named helpers
  (`invalidateAfterSolve()`, `invalidateAdminTeam(id)`).
- `lib/forms/use-api-form.svelte.ts`: same shape, two fixes — submit-time
  validation maps issues per-field (not one `_form` blob), and success
  detection checks `route.goodResponses` kinds instead of
  `kind.startsWith('good')`.
- Root layout boot: block on `clientConfig` fetch, prefetch `userSelf`,
  init analytics, mount toast host — same as today.
- Admin challenge editor: `$state` class `ChallengeEditor` with
  `mode: 'idle' | 'viewing' | 'editing' | 'creating' | 'saving' | 'confirmDiscard' | 'confirmDelete' | 'deleting'`,
  `dirty` derived (JSON compare vs `originalForm`), and the same behavioral
  contract (unsaved-changes interception, form snapshot/restore).

## Admin filter framework

Build ONCE, from the submissions page's generalized design (research 04 §7):
`MultiFilter` core + `ValueFilterFamily` descriptor contract + shared table
shell (pinned toolbar, sticky sortable header, virtualized rows, fingerprint
effects, empty states). Teams' status/division become two families; pseudo
options (`unverified`) and the time-range family are first-class variants.
The old app wrote this twice; we write it once, before either admin table.

## Conventions

- **Formatting**: repo prettier (no semicolons, single quotes, 2-space,
  `printWidth: 100` for `.svelte`); imports sorted by
  `@ianvs/prettier-plugin-sort-imports`.
- **TypeScript**: root tsconfig strictness (`strict`,
  `noUncheckedIndexedAccess`, `verbatimModuleSyntax`, …); `svelte-check`
  zero-warning policy from day one; `bun run typecheck` wired into the
  workspace filter.
- **Imports**: `$lib/...` absolute only; `type` imports explicit.
- **Files**: `kebab-case` for everything (matches apps/web); components flat
  within their feature directory; one concern per lib module.
- **Props**: one `type Props = {...}` per component, destructured with
  defaults; boolean attributes forwarded as `attr={value || undefined}`.
- **Comments**: near-zero; permitted only for non-obvious constraints
  (e.g. Utopia generator URLs, the Zag thunk rule at wrapper sites).
- **JS↔CSS contract**: JS computes state; CSS renders it. Toggle data
  attributes / set custom properties — never inline styles or classList.
- **Routes are thin**: pages fetch via query options and compose feature
  components; feature components live next to their route.
- Svelte MCP (`svelte-autofixer`) on every new/edited component before commit.

## Verification

Per-phase gates are in PLAN.md. Standing policy: `bun run --filter '@rctf/web-new' check`
